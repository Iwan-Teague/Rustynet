# Release Signing Runbook (Windows Authenticode)

Last updated: 2026-04-28

## 1) Mission

Every Rustynet Windows release artefact (`rustynetd.exe`) ships
Authenticode-signed. The signature is the trust anchor the W2.1b
chain validator (`crates/rustynetd/src/windows_authenticode.rs` +
`rustynet_windows_native::verify_authenticode_chain`) gates daemon
startup against; an unsigned or self-signed artefact gets
`TRUST_E_NOSIGNATURE` / `CERT_E_UNTRUSTEDROOT` and the daemon
refuses to start.

This runbook captures the operator-facing setup. The reviewed
workflow + helper landed in:
- [.github/workflows/release-windows.yml](../../.github/workflows/release-windows.yml)
- [scripts/release/Sign-RustyNetWindowsBinary.ps1](../../scripts/release/Sign-RustyNetWindowsBinary.ps1)

## 2) Prerequisites

Before tagging a release the operator MUST have configured one of
the supported signing routes (see Â§3) AND committed the resulting
GitHub Secrets to the repository's secret store.

The post-sign verification step in the workflow runs the daemon's
own W2.1b chain validator against the freshly signed artefact, so
a misconfigured cert (untrusted root, missing intermediate, expired,
revoked) fails the workflow before the artefact uploads to the
GitHub release.

## 3) Supported Signing Routes

### 3.1 PFX File on Disk (simplest, baseline)

A code-signing certificate exported as a `.pfx` (PKCS#12) file with
a passphrase. Acceptable for non-production / staging signing AND
for production signing IF the PFX is sourced from a hardware-token-
backed export (HSM-issued key, PFX-wrapped for transport).

**NOT** acceptable: a soft-key PFX generated by `openssl req` and
self-signed. Such a key fails W2.1b's `WinVerifyTrust` check
(`CERT_E_UNTRUSTEDROOT`) by design.

**Setup:**

1. Acquire a code-signing cert from a Microsoft-trusted CA (DigiCert,
   Sectigo, GlobalSign, etc.). The certificate MUST chain to a root
   in the default Windows root store.
2. Export to PFX with a strong passphrase (minimum 32 random bytes,
   stored only in a password manager + the GitHub Secrets store).
3. Base64-encode the PFX:
   ```bash
   base64 -w0 release.pfx > release.pfx.b64
   ```
4. Configure the GitHub Secrets in the repository (Settings â†’ Secrets
   and variables â†’ Actions â†’ New repository secret):
   - `RUSTYNET_SIGNING_PFX_BASE64` = contents of `release.pfx.b64`
   - `RUSTYNET_SIGNING_PFX_PASSWORD` = the PFX passphrase
5. Optional: `RUSTYNET_SIGNING_TIMESTAMP_URL` = your preferred RFC
   3161 timestamp authority. Defaults to
   `http://timestamp.digicert.com` if unset.

The release workflow verifies BOTH secrets are non-empty BEFORE
attempting any sign step; missing secrets fail the job loud.

### 3.2 Azure Key Vault (recommended for production)

The signing key never leaves Azure Key Vault. The release workflow
authenticates to Key Vault via OIDC federation + retrieves a
short-lived token, then either:
- exports a sign-only PFX session token (NOT supported by signtool
  today; requires a wrapper like AzureSignTool), OR
- (recommended) uses `AzureSignTool` (https://github.com/vcsjones/AzureSignTool)
  which fetches the cert reference from Key Vault and signs without
  ever materialising a PFX on the runner.

Setting up AzureSignTool is OUT OF SCOPE for the v1 runbook (the
PFX flow above is what the workflow currently exercises). Tracked
as a follow-up in
[documents/operations/active/SecurityHardeningAudit_2026-04-28.md](active/SecurityHardeningAudit_2026-04-28.md)
under a future "release signing route hardening" entry.

### 3.3 Hardware Token (Yubikey, SafeNet eToken)

Same PFX flow as Â§3.1 but the PFX is exported from the hardware
token's PKCS#11 / PKCS#12 interface as a *transport* PFX (the
private key still lives on the token; only the cert + a wrapped
export are in the PFX).

Operationally identical to Â§3.1 from the workflow's perspective.

## 4) Release Tag Convention

The workflow triggers on tags matching `v*.*.*` (e.g. `v0.1.0`,
`v0.2.3-rc1`). A `git push origin v0.1.0` after `git tag -a v0.1.0`
runs the release-signing job exactly once.

Manual `workflow_dispatch` invocation supports a `dry_run` input
(default `true`) that builds without signing â€” used for path-
validation testing on a branch that does not yet have signing
secrets configured.

## 5) Verification (post-release)

A consumer of a Rustynet release MUST verify the signed binary's
chain before installing. Two independent verification paths:

1. Built-in: install the binary, then run
   ```
   "C:\Program Files\RustyNet\rustynetd.exe" windows-authenticode-check
   ```
   The W2.1b chain validator returns `overall_ok=true` only when
   `WinVerifyTrust` reports `Verified` end-to-end.

2. Out-of-band: PowerShell
   ```powershell
   Get-AuthenticodeSignature C:\Path\To\rustynetd.exe
   ```
   Status MUST be `Valid`. Any other status (`UnknownError`,
   `NotSigned`, `HashMismatch`, `NotTrusted`) is fail-closed and
   the binary MUST NOT be installed.

The release workflow ships a `rustynetd.exe.sha256` alongside the
signed binary; consumers SHOULD also verify the SHA-256 against the
published release notes' expected hash before running the
Authenticode check.

## 6) Rollback Procedure

If a signed release ships and is later found to contain a defect
that requires recall:

1. **Do NOT revoke the signing certificate** unless the private key
   is suspected compromised. Revocation invalidates ALL prior
   signed releases, breaking every deployed install.
2. Publish a higher-version release with the fix. The W2.1b
   validator does not pin to a specific version; consumers updating
   to the new release pass the chain validator on the new artefact.
3. Mark the bad release in the GitHub Releases UI as deprecated
   with a clear "do not install â€” see CVE-XXXX" notice in the
   release notes.

If the private key IS suspected compromised:

1. Revoke the certificate at the issuing CA.
2. Wait for revocation to propagate to CRL/OCSP responders (24-48h
   typically).
3. The W2.1b validator's `WTD_REVOCATION_CHECK_CHAIN_EXCLUDE_ROOT`
   posture surfaces revoked-cert binaries as
   `CERT_E_REVOKED`/`CRYPT_E_REVOKED` after propagation; deployed
   installs will refuse to restart with the revoked-signed binary.
4. Re-issue the cert from the CA, re-configure the GitHub Secrets,
   and ship a new release with the fresh cert chain.

## 7) Threat Model Notes

- **PFX exfiltration from GitHub Secrets**: Mitigated by the OIDC /
  Key Vault upgrade path in Â§3.2 (recommended for production). The
  v1 PFX-on-disk flow trusts GitHub's secret-store integrity.
- **Workflow log leakage of signing material**: The helper script
  reads the PFX password from a *named env var* rather than from
  the command line, so the password does not appear in CI logs.
  Workflow steps that reference the PFX/password DO have the
  secret values masked by GitHub Actions' secret-redaction layer.
- **Replay of an old signed binary**: Mitigated by the daemon's
  own integrity verification (the chain validator confirms the
  certificate is still valid + not revoked at runtime, on every
  daemon start). An attacker who replays an old signed binary with
  a since-revoked cert gets `CERT_E_REVOKED` at startup.
- **Misconfigured timestamp authority**: A signed binary without a
  countersigned timestamp expires when the signing cert expires
  (typically 1-3 years). The helper script's `signtool /tr` flag
  is REQUIRED (the script rejects an empty timestamp URL); a
  countersigned timestamp lets the binary remain valid past the
  signing cert's own expiry. Defaults to DigiCert's public RFC
  3161 endpoint.
