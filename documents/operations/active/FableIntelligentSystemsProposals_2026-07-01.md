## FIS-0001: Continuous path-quality EWMA scoring + bounded make-before-break migration
- **Pipeline / area:** rustynetd traversal/dataplane control loop (crates/rustynetd/src/traversal.rs, phase10.rs, daemon.rs) + rustynet-backend-wireguard userspace-shared engine + rustynet-backend-api trait.

- **Current behavior (grounded):**
  `boringtun::noise::Tunn::stats()` (third_party/boringtun/src/noise/mod.rs:577-585) already computes `loss: f32` — an EWMA over the last `N_SESSIONS=8` rekey sessions with decaying weights 9.0/3.0/1.0/... (`estimate_loss`, mod.rs:543-571) — and `rtt: Option<u32>` in milliseconds, the wall-clock delta between handshake-init send and authenticated handshake-response receipt (`noise/handshake.rs:315,646-647`). Both values are computed and then discarded: `crates/rustynet-backend-wireguard/src/userspace_shared/engine.rs:345` destructures `let (_handshake, peer_tx, peer_rx, _loss, _rtt) = peer_state.tunnel.stats();` and a second call at line 531 does the same. `TunnelStats` (crates/rustynet-backend-api/src/lib.rs:105-110) has no rtt/loss/jitter field and is engine-wide, not per-peer. Command-only backends (linux_command.rs:453-461, macos_command.rs:662-670, windows_command.rs:518-528) never construct a `Tunn` in-process, so they have zero RTT/loss signal of any kind — this proposal is Linux/macOS-userspace-shared-backend-only by construction, not a cross-backend feature.
  The only signal that reaches path decisions is `peer_latest_handshake_unix` (seconds resolution, `rustynet-backend-api/src/lib.rs:184-185`). `traversal_probe_due` (crates/rustynetd/src/daemon.rs:5915-5949) only re-races a `PathMode::Direct` peer when its current endpoint drops out of the signed candidate set OR `traversal_handshake_is_fresh` (daemon.rs:6037-6040, `age <= 30s`) fails; `PathMode::Relay` reprobes every `DEFAULT_TRAVERSAL_PROBE_REPROBE_INTERVAL_SECS=30` (traversal.rs:58). A healthy Direct winner is never re-scored against other candidates. `TraversalSession` (traversal.rs:995-1100) — including `on_endpoint_roamed` (1061-1079) and `consecutive_direct_failures`/`relay_switch_after_failures` (999, config field, default 3) — has zero production callers; the daemon's own `traversal_probe_due`/reconcile logic is the real policy. Endpoint programming itself is a single-slot rewrite: `TunnelBackend::update_peer_endpoint(node_id, endpoint)` (backend-api/src/lib.rs:173-177) and `Phase10Controller::reconfigure_managed_peer` (phase10.rs:5889-5909) both mutate one `SocketEndpoint` per `NodeId` — WireGuard has exactly one live conversation per peer key, so probing an alternate candidate today means rewriting the *current* peer's endpoint, not opening a second path alongside it.

- **Proposed mechanism:**
  Named technique: **EWMA/CUSUM path-quality scoring** (classic change-detection, e.g. Page 1954 CUSUM, Roberts 1959 EWMA control charts) feeding a **hysteresis-gated make-before-break handoff**, structurally modeled on QUIC connection migration's probe-before-commit pattern (RFC 9000 §9: PATH_CHALLENGE/PATH_RESPONSE validate a new path before traffic moves) and Tailscale's continuous direct-vs-DERP re-evaluation (magicsock periodic re-STUN + best-path reselection).

  Concrete Rustynet adaptation, three data-structure/wire changes:
  1. **Per-peer `PathQualitySample`** (new struct in traversal.rs, engine-local, never serialized/signed): `rtt_ewma_ms: f32`, `rtt_var_ewma_ms: f32` (Jacobson/Karn-style smoothed RTT + mean deviation, RFC 6298 form: `rtt_ewma += 0.125*(sample - rtt_ewma); rtt_var_ewma += 0.25*(|sample - rtt_ewma| - rtt_var_ewma)`), `loss_ewma: f32` (read directly from boringtun's existing `estimate_loss()`, not re-derived), `consecutive_stale_handshakes: u8`, `last_updated_unix: u64`. Populated by un-discarding `_loss`/`_rtt` at the two existing `engine.rs` call sites and threading them through a new `TunnelBackend::peer_path_quality(&mut self, node_id) -> Result<Option<PathQualitySample>, BackendError>` trait method with a default `Ok(None)` (so command-only backends compile unchanged and fail closed to "no quality data" — see constraint check). Composite score: `quality = w_rtt * clamp(1 - rtt_ewma/RTT_CEIL_MS, 0, 1) + w_jit * clamp(1 - rtt_var_ewma/JITTER_CEIL_MS, 0, 1) + w_loss * clamp(1 - loss_ewma/LOSS_CEIL, 0, 1)`, weights `w_rtt=0.4, w_jit=0.2, w_loss=0.4` (loss weighted equal to RTT because loss bursts are the sharper WireGuard-handshake-retry driver). Score is `None` (fail-closed, no recommendation) until at least 2 real samples exist — matches constraint #2's "default to no-op when inputs are missing."
  2. **Candidate re-score trigger without disturbing the active path**: extend `traversal_probe_due` with a *separate, lower-frequency* comparative-check timer (`quality_reconsider_interval_secs`, proposed default 300s — 10x the existing 30s reprobe, so it never competes with the existing liveness cadence) that fires only when (a) the active path is `Direct` and healthy per existing freshness check, AND (b) a same-priority-class alternate candidate exists in the last-known signed candidate set. On fire, it does **not** touch the live peer endpoint. Instead it schedules a bounded **shadow probe**: a single unauthenticated STUN-style UDP echo (reusing the existing `build_stun_binding_request`/raw-UDP machinery already in traversal.rs, not a new protocol) sent from the *same* authoritative socket to the *candidate* endpoint, with a 200ms timeout, purely to confirm L3 reachability + coarse RTT — this is NOT a WireGuard handshake and does not touch the peer's configured endpoint, so the live tunnel is fully undisturbed during evaluation (unlike today's `execute_ice_pair_race`, which always flips the live endpoint to probe). This is the honest gap: it validates reachability+RTT of the alternate, but not a real WireGuard handshake RTT/loss on that path — see cost/tradeoffs.
  3. **Hysteresis-gated commit, still single-slot**: only if the shadow-probed candidate's *estimated* quality exceeds the active path's measured `quality` by a switch margin (`SWITCH_MARGIN = 0.15`, ~15 percentage points of composite score — chosen to be well above the score's own noise floor, since `rtt_var_ewma` alone can shift the score several points tick-to-tick) AND the active path has NOT switched in the last `dwell_time_secs` (proposed 180s, ~6x the quality_reconsider_interval so at most one switch per ~5 reconsider windows) does the daemon perform the actual migration: `reconfigure_managed_peer` to the new endpoint (the existing single-slot rewrite — no new wire format), immediately followed by one `initiate_peer_handshake(force_resend=true)`. Because the shadow probe already confirmed L3 reachability, the handshake round-trip that follows the flip is expected to succeed in one attempt instead of the current blind flip-and-hope; if it does NOT complete within `handshake_freshness_secs` (existing 30s constant), the *existing* `traversal_probe_due`/re-race logic (unmodified) takes over and either reverts to the prior candidate (still in the signed set) or falls to relay — i.e., failure recovery reuses 100% of the existing FSM, this proposal adds zero new failure-recovery code paths. No signed-state round-trip occurs mid-migration because the candidate set, priorities, and relay fallback are unchanged — only the choice of *which already-signed-valid candidate* to program changes, which is exactly the discretion `execute_ice_pair_race` already exercises today, just re-run less blindly.

- **Why this is the intelligent version:**
  At 2-3 node scale the win is genuinely marginal — with one or two peers there's rarely a *better* alternate candidate sitting idle, and the 30s handshake-freshness reprobe already catches outright breakage. The payoff shows up at the higher end of the target range (10-50 nodes, mixed home/mobile/anchor topology): today a suboptimal-but-technically-fresh Direct pick (e.g., a double-NAT srflx path that happens to have punched through before a cleaner LAN-host path became visible, or a laptop that roamed onto worse Wi-Fi and stayed "fresh" because keepalives still land) is *never* reconsidered — it is locked in until it goes fully stale. Reusing boringtun's already-computed loss/RTT (zero new crypto, zero new measurement primitive — literally un-discarding two `let _ = ...` bindings) converts "still breathing" into "how well is it breathing," which is the exact gap the grounding notes identify. The shadow-probe-before-flip step is the one truly new mechanism, and it is small and bounded — it turns today's `execute_ice_pair_race` "flip and hope" into "flip only after independent reachability evidence," a real reliability improvement to the *existing* re-race path even independent of the new scoring.

- **Cost / tradeoffs:**
  Wire bytes: zero new signed-bundle fields; the shadow probe is a raw UDP packet (~28 bytes, same as existing STUN) sent peer-to-peer over the mesh's own authoritative socket, not the public STUN infra — new but tiny per-mesh-hour traffic (at most 1 probe per peer per `quality_reconsider_interval_secs`=300s while Direct and healthy, i.e. ≤12/hour/peer, only when an alternate candidate exists). CPU/memory: EWMA update is O(1) float ops per handshake event, `PathQualitySample` is ~40 bytes per peer — negligible even on Raspberry-Pi-class anchors handling 50 peers (2KB). New failure modes: (a) a spoofed/racing UDP echo response from an off-path attacker could inflate a fake candidate's apparent quality, but it can only ever nominate a candidate that is *already in the current signed traversal-candidate set* — the shadow probe is a filter for "should we bother re-flipping," never a source of new trust, so worst case is a wasted flip-and-revert (bounded by dwell time + existing 30s recovery), not a security compromise; still needs a note that response validation should check source-address match to avoid trivially widening this from "wasted flip" to "attacker can force flap on a timer" (mitigated by `dwell_time_secs` capping flap frequency). (b) hysteresis mistuning risk: if `SWITCH_MARGIN`/`dwell_time_secs` are too tight, this reintroduces exactly the flapping the direction brief warns about; both are cheap to tune via a bench harness before any live-lab exposure (see falsification below). Operational complexity: one new trait method with a safe default, one new engine-local (non-persisted, non-signed) struct, and an additive timer alongside — not a rewrite of `execute_ice_pair_race` or the transport FSM.

- **Constraint check:**
  1. No custom crypto/VPN protocol — PASS: scoring reads existing boringtun-computed values; migration reuses the existing `update_peer_endpoint`+handshake calls verbatim; the shadow probe reuses the existing raw-UDP STUN-style request builder, not a new protocol.
  2. Default-deny/fail-closed — PASS: `peer_path_quality` defaults to `Ok(None)` for backends that can't provide it (command-only backends behave exactly as today — no regression, no silent optimism); a switch requires 2+ real samples, a margin threshold, AND dwell-time — absence/staleness of any input means "keep current path," never "switch." No capability or trust changes — only *which already-signed-valid endpoint to program* changes, same discretion the existing prober already has, so no new admin-signed surface is needed.
  3. Rust-first, no new runtime category — PASS: pure Rust arithmetic (EWMA/threshold comparisons), no ML runtime, no external service.
  4. Backend-abstraction boundary — PASS: `PathQualitySample`/scoring lives in rustynetd (daemon-side control logic), not in rustynet-control/rustynet-policy; the new trait method is on `TunnelBackend` in rustynet-backend-api (the correct abstraction layer) and returns an abstract struct, no boringtun types leak upward.
  5. Resource profile (Pi-class anchor, 2-50 nodes) — PASS: O(1) per-peer state, sub-KB memory, probe traffic bounded to ≤12/peer/hour and only while already healthy; explicitly NOT a datacenter mechanism (no ML inference, no per-packet telemetry pipeline, no time-series DB).

- **Incremental build path:**
  - Phase 1 (S, <half day): un-discard `_loss`/`_rtt` in `engine.rs`, add `peer_path_quality` to `TunnelBackend` with `Ok(None)` default, thread through userspace-shared engine only. No behavior change — pure plumbing + unit tests asserting command-only backends still return `None`.
  - Phase 2 (M, 1-2 days): implement `PathQualitySample` + EWMA update functions in traversal.rs as pure, independently unit-testable functions (given a sequence of (rtt, loss) samples, assert score trajectory) — no daemon wiring yet, so this phase is entirely bench/unit-test verifiable before touching the reconcile loop.
  - Phase 3 (M, 1-2 days): wire the `quality_reconsider_interval_secs` timer + shadow-probe-only path into `traversal_probe_due`/reconcile — emit a structured log/metric of "would-switch" recommendations WITHOUT actually calling `reconfigure_managed_peer` (dry-run/shadow mode first, matching this repo's FAIL-LOUD live-stage philosophy of proving the signal before acting on it).
  - Phase 4 (L, needs a design decision + live-lab sign-off): flip shadow-mode to live migration behind a config flag, default off; add a live-lab stage (e.g. `validate_path_quality_migration`) that induces asymmetric loss/latency on one candidate path (netns tc-netem, already used elsewhere per the NAT-profile lab plan) and asserts the mesh migrates to the better candidate within `dwell_time_secs + quality_reconsider_interval_secs` without a handshake gap exceeding `handshake_freshness_secs`.

- **How you would know it worked / what would falsify it:**
  Positive signal: a live-lab stage that impairs the currently-active Direct candidate (inject loss/jitter via tc-netem on that specific path while a cleaner alternate candidate remains reachable) shows the daemon switching to the better candidate within one `quality_reconsider_interval_secs` + `dwell_time_secs` window, with zero relay fallback and a bounded handshake gap. Falsification: if in practice `rtt_var_ewma`/`loss_ewma` noise on ordinary home Wi-Fi/cellular is large enough that no `SWITCH_MARGIN` avoids either (a) flapping between two similar-quality candidates or (b) never triggering because real-world quality differences are smaller than measurement noise, the mechanism is not worth the complexity — the falsifying result is a live-lab run showing >1 switch per hour on a stable network with no actual path degradation, which would mean this candidate should be shelved in favor of just tightening the existing 30s freshness/reprobe constants.

- **Prior art:** QUIC connection migration, RFC 9000 §9 (Iyengar & Thomson, 2021) — PATH_CHALLENGE/PATH_RESPONSE validate-before-use pattern this proposal's shadow probe mirrors. Tailscale magicsock continuous direct-vs-DERP re-evaluation (Tailscale engineering blog, "How Tailscale works," 2020-2023). Jacobson & Karels, "Congestion Avoidance and Control," 1988 / RFC 6298 — smoothed RTT + mean-deviation EWMA form reused verbatim for `rtt_ewma`/`rtt_var_ewma`. Page, "Continuous Inspection Schemes," Biometrika 1954 (CUSUM) and Roberts, "Control Chart Tests Based on Geometric Moving Averages," Technometrics 1959 (EWMA control charts) as the classical change-detection basis cited in the direction brief. MPTCP path management (RFC 8684) as the broader precedent for maintaining multiple scored candidate paths per logical connection, though Rustynet's single-peer-endpoint WireGuard constraint (unlike MPTCP's native multi-subflow support) is exactly why this proposal cannot offer true dual-live-path make-before-break and instead uses probe-then-flip.

- **Adversarial verification (independent citation fact-check + constraint review, 2026-07-01):** Citation fact-check **PASS** — all load-bearing current-code claims verified against the repo (`Tunn::stats()` at `third_party/boringtun/boringtun/src/noise/mod.rs:577-585`, `estimate_loss()` at `mod.rs:543-571`, `Handshake::last_rtt` set at `handshake.rs:646-647`, the two discarded `(_loss, _rtt)` destructures at `engine.rs:345/531`, `TunnelStats` lacking rtt/loss fields, `traversal_probe_due` never re-scoring a healthy Direct path, `TraversalSession` having zero non-test callers). Constraint review verdict: **accept-with-fixes** — fixes to fold in before any phase-1 work:
  - Drop the CUSUM citation or implement an actual CUSUM statistic: the mechanism as designed is EWMA + fixed-margin threshold + dwell-time hysteresis and should be described as exactly that.
  - Promote spoofed shadow-probe response validation (source-address-bound response matching + an echoed per-probe nonce) from a cost-section prose note to a mandatory, numbered acceptance criterion in phases 3/4 — an off-path attacker forging probe responses to inflate a candidate's score and force flap-on-a-timer must be structurally excluded, not advisorily noted.
  - State explicitly that `PathQualitySample` / the new `peer_path_quality` return type carry only primitive fields (f32/u8/u64) with zero reference to `boringtun::noise` types, closing the backend-boundary-leak question definitively.
  - Clarify that RFC 6298 is borrowed only for its SRTT/RTTVAR smoothing constants (alpha=1/8, beta=1/4), not its RTO/backoff semantics.
  The reviewer independently confirmed distinctness from planned D14 work: D14.a-f are pre-connection establishment (port-mapping, NAT classification, punch timing); this proposal is post-establishment path re-scoring — a different lifecycle stage, not a restatement of planned work.
- **Designer's own weakest point (disclosed):** true zero-blip make-before-break is not achievable at today's backend boundary — `TunnelBackend::update_peer_endpoint` (`crates/rustynet-backend-api/src/lib.rs:173`) rewrites the single peer-endpoint slot, so holding two live WireGuard conversations to the same peer requires a trait extension; the honest guarantee as scoped is bounded-blip migration, not zero-blip.

## FIS-0002: Machine-checked formal spec of membership apply/anti-replay/rollback protocol

- **Pipeline / area:** rustynet-control membership state machine (`crates/rustynet-control/src/membership.rs`) — the signed-bundle apply/replay/rollback core consumed by `rustynetd`'s `handle_membership_apply` (daemon.rs:7695) and bootstrap replay (`load_verified_membership`, daemon.rs:3990).

- **Current behavior (grounded):** `MembershipState { schema_version: u8, network_id: String, epoch: u64, nodes: Vec<MembershipNode>, approver_set: Vec<MembershipApprover>, quorum_threshold: u8, metadata_hash: Option<String> }` (membership.rs:160-168, read directly). `apply_signed_update` (membership.rs:693-737, read directly) runs a fixed 8-check gate in order: `state.validate()` → `record.canonical_payload()` (which itself enforces `epoch_new == epoch_prev+1` and `created_at < expires_at`) → network_id match → expiry (`now > expires_at` → `Expired`) → future-dating (`created_at > now+90s` → `FutureDated`, `MEMBERSHIP_CLOCK_SKEW_SECS=90`) → `prev_state_root == state.state_root_hex()` → epoch chain (`epoch_prev == state.epoch && epoch_new == state.epoch+1`) → `verify_membership_signatures` (quorum + owner-role gate) → `reduce_membership_state` using the record's own `created_at_unix` (the RSA-0009 determinism fix, documented at the call site) → `next.validate()` → computed root == `record.new_state_root` → `replay_cache.observe(update_id, epoch_new)` (membership.rs:552-562, read directly: rejects a seen `update_id` OR `epoch_new <= max_epoch`) applied LAST, after every other check succeeds. `replay_membership_snapshot_and_log` (membership.rs:906-932, read directly) re-derives state from a snapshot + hash-chained log on every daemon bootstrap, seeding `replay_cache.max_epoch = snapshot.epoch` and requiring the first replayed entry's `epoch_prev`/`prev_state_root` to match the snapshot exactly. This entire 8-operation, epoch-chained, dual-layer-replay-protected state machine is currently proven ONLY by ~40 named unit tests in `membership.rs` (lines 1944-3480) plus 4 integration tests in `rustynetd/tests/membership_replay_protection.rs` — confirmed by repo search: no `proptest`, `stateright`, `kani`, or `.tla` file exists anywhere in the workspace (`Cargo.lock` has zero `proptest` entries; no `*.tla`/`*.cfg` files found repo-wide). Every invariant the reducer holds is implicit in hand-picked test cases, not a checked universal property.

- **Proposed mechanism:** A **TLA+ model** (Lamport, *Specifying Systems*, 2002; methodology per Newcombe et al., *How Amazon Web Services Uses Formal Methods*, CACM 2015) of exactly the state actually in the struct — `epoch`, `nodes` (as a function `node_id -> {status, capabilities}`), `approver_set` (as `approver_id -> {role, status}`), `quorum_threshold`, plus the two replay-detection variables that live outside `MembershipState` proper: `ReplayCache` (`seen_ids`, `max_epoch`) and `Watermark` (`epoch`, `state_root`) from `membership_watermark_is_replay` (daemon.rs:11939). The adaptation is concrete: model `Next` as the disjunction of the 8 `MembershipOperation` variants (membership.rs:341-364) each guarded by its actual reducer precondition (e.g. `AddNode` guarded by `node_id \notin DOMAIN nodes`, `RevokeNode` guarded by `nodes[node_id].status = Active`, `SetNodeCapabilities` guarded by `nodes[node_id].capabilities # {BlindExit}` mirroring the reducer's line-1179 guard verbatim) — not a generic "distributed KV store" template. **Three genuine properties, written in real TLA+ syntax:**

  ```tla
  \* P1: Epoch strictly increases by exactly 1 per accepted transition (no skip, no replay-at-same-epoch)
  EpochMonotonicity == [][epoch' = epoch \/ epoch' = epoch + 1]_epoch

  \* P2: No update_id is ever accepted twice (replay_cache.observe semantics, membership.rs:552-562)
  NoDoubleApply ==
      \A id \in SeenIds : [](id \in SeenIds => [](id \in SeenIds))
      \* stronger form actually checked: an update accepted at epoch e can never be
      \* re-accepted at any epoch e' <= max_epoch_at_acceptance_time
      /\ [][\A u \in AppliedUpdateIds' \ AppliedUpdateIds :
             u \notin AppliedUpdateIds]_AppliedUpdateIds

  \* P3: A Revoked node's stale pre-revocation bundle can never re-apply post-revocation
  \* (prev_state_root chain forces any competing branch to be rejected, membership.rs:713-714)
  RevocationDominatesStaleBundle ==
      \A n \in DOMAIN nodes :
          nodes[n].status = "Revoked" =>
              [](\A upd \in PendingUpdates :
                   upd.target = n /\ upd.prev_root # StateRoot(state) => ~ENABLED Apply(upd))
  ```
  These are exactly the three candidates named in the brief, written against the model's own operators (`StateRoot`, `Apply`), not restated English. A 4th property worth adding once the first three are running: `OwnerNeverEmpty == [](\E a \in DOMAIN approvers : approvers[a].role = "Owner" /\ approvers[a].status = "Active")` — this one is interesting precisely because it currently is **not** an invariant of the real code (see Gaps: no guard prevents quorum-revoking the last Owner), so TLC would find the counterexample trace on the first run, which is exactly the kind of result that justifies the exercise.

- **Why this is the intelligent version:** At 2-3 node scale the unit-test suite already covers the reachable state space by hand, and this buys almost nothing — the honest reading of the current 40-test suite is that it is not obviously deficient for a 2-3-node mesh reasoned about by one operator. The real payoff is at the **8-operation × N-node × M-approver combinatorial surface** the tests do NOT enumerate: interleavings across operation types (revoke-then-restore-then-set-capabilities in different orders), the specific u8-truncation-at-256-approvers bug and last-Owner-deadlock bug this grounding pass just found by reading code, not by running any test — both are exactly the class of "boundary condition nobody wrote a test for" that model checking finds mechanically and testing finds by luck. TLC exhaustively explores the *reachable* state graph for a bounded model (e.g. 4 nodes, 3 approvers, quorum=2) in minutes, which is intractable to hand-enumerate as unit tests but perfectly tractable as a bounded-model-check — this is the same argument Newcombe et al. make for AWS S3/DynamoDB: small models, big payoff on protocols with many interacting state transitions and adversarial timing, which is precisely what an epoch-chained multi-op quorum reducer is.

- **Cost / tradeoffs:** Zero runtime cost — this is 100% off the critical path, no wire format change, no new struct field, no daemon code touched. The cost is entirely engineering/maintenance: (a) TLA+/TLC toolchain (Java-based `tla2tools.jar`) is a new *development-time* dependency, not a runtime one — install once, run in CI as a separate job, never ships in any binary; (b) the model is a **second description of the same logic**, and the single biggest risk in this whole proposal is spec rot — if the Rust reducer changes (e.g. a 9th operation is added) and nobody updates the TLA+ `Next` disjunct, the spec silently stops meaning anything while still "passing." This is the load-bearing risk the keep-in-sync plan below has to actually solve, not gesture at. (c) model-checking state-space cost is real: naively modeling `Vec<MembershipNode>` unbounded is infinite-state; the model must be bounded (small fixed node/approver counts) to run TLC at all, which means the model checker proves properties for the *small* instance and the proptest/stateright layer is what has to carry confidence to realistic N — this is a genuine gap between "TLC says property P holds for 4 nodes" and "P holds for the 50-node target scale," and the proposal must not overstate what TLC alone buys. (d) new attack surface: none — nothing in this proposal ships to any node or touches the trust boundary.

- **Constraint check:**
  1. No custom crypto/VPN protocol — PASS. Zero touch to `rustynet-backend-api`, WireGuard, or signature primitives; models `verify_membership_signatures` as an opaque `SigValid(update)` boolean abstraction (see model boundary below), never re-derives or replaces it.
  2. Default-deny/fail-closed structural — PASS, and this proposal *strengthens* the argument for it: model checking is specifically suited to proving "on missing/stale/ambiguous input, the only enabled transition is deny," and would be the mechanism that catches a future regression that accidentally makes an ambiguous case permissive. No elevated capability is granted by this work; it is pure verification tooling with no runtime code path.
  3. ## FIS-0003: Digest-based anti-entropy + bounded-fanout epidemic dissemination for signed gossip bundles

- **Pipeline / area:** Peer-endpoint candidate gossip (`crates/rustynetd/src/peer_gossip.rs` + `crates/rustynetd/src/gossip_runtime.rs`). Endpoint-reachability candidates only — the signed MEMBERSHIP bundle (anchor bundle-pull, `crates/rustynet-control/src/membership.rs`) is a structurally separate distribution mechanism and is addressed only as an optional Phase 4 extension, not the primary target.

- **Current behavior (grounded):** `GossipNode::maybe_mint_and_broadcast` (gossip_runtime.rs:278) re-mints a signed `GossipBundle` every `GOSSIP_REMINT_INTERVAL_SECS` = 30s (line 72) or on candidate change, and on every mint pushes the full bundle to **every** registered peer via `ordered_peer_ids_for_rebroadcast(None, None)` (line 479) — an unconditional O(N) push per mint, O(N²) mesh-wide per remint cycle. `ingest_inbound_bundle` (line 376) does the same on every *accepted* inbound bundle: re-push to every known peer except the sender and originator (line 448-462), so an epidemic re-flood happens on every hop with no TTL/hop-count anywhere in the wire format (confirmed: `GossipBundle` carries only `source_node_id`, `sequence`, `timestamp_unix`, `candidates`, `signature` — peer_gossip.rs:106-112). The only flood damper is the strict-monotonic per-source sequence check in `accept_bundle_with_now` (peer_gossip.rs:470, rejects via `SequenceNotMonotonic` at line 513): a duplicate is dropped and NOT re-pushed, but a first-arrival is always fully re-flooded regardless of how many peers already plausibly have it. There is no digest/summary/ACK/pull-repair primitive anywhere in the module — grep confirms the entire `GossipNode` API surface is push-only (`register_peer`, `maybe_mint_and_broadcast`, `ingest_inbound_bundle`, watermark I/O). **Production-attachment gap (verified):** `attach_gossip_runtime` (daemon.rs:5003) has zero callers anywhere in the workspace (`grep -rn "attach_gossip_runtime" crates/` returns only its own definition, a comment at daemon.rs:3948, and an MCP lab-state string literal at `rustynet-mcp/src/bin/lab_state.rs:4377` reading `"gossip is a wired-but-not-yet-daemon-integrated subsystem"`) — this entire mechanism is currently test-only (`tests/gossip_three_peer_mesh.rs`); this proposal is designed to slot into that frame once D2.7 wiring lands, not to alter already-shipped production behavior.

- **Proposed mechanism:** **Plumtree** (Leitão, Pereira, Rodrigues, "Epidemic Broadcast Trees," SRDS 2007) supplies the shape: an implicit eager-push spanning tree for the common case plus lazy `IHAVE`/`GRAFT` digest-based repair for the tail, self-healing without global topology knowledge. **Rustynet-specific adaptation** (this is where the concrete change lives, not a restatement of Plumtree):
  1. **New lightweight digest message**, `GossipIHave { source_node_id: [u8;32], sequence: u64 }` — 40 bytes (32+8), no candidate payload, no signature (rationale below). Wire version bumped to a new variant tag inside the existing 4 KiB-capped UDP channel (same port 51821, same transport). This is NOT a new signed artifact type; it is an unsigned *hint* that never causes state mutation on its own — see constraint check.
  2. **Peer link state becomes eager/lazy per (peer, source)** pair rather than global: `GossipNode` gains a small `HashMap<[u8;32] /* peer_id */, PeerLinkMode>` where `PeerLinkMode` starts `Eager` for every registered peer (matches today's behavior exactly — zero behavior change at bootstrap or ≤~8 peers) and demotes to `Lazy` only after that peer has independently ACK'd (via inbound bundle from that peer carrying a `>=` sequence, i.e. it already has current state) redundant delivery of the same `(source_node_id, sequence)` tuple N times (default N=1: first redundant delivery demotes). Demotion is purely a local optimization hint — reversible, and a `Lazy` peer that stops sending `IHAVE`s for a source gets promoted back to `Eager` for that source after a bounded silence window (self-healing, matches Plumtree's tree-repair property).
  3. **Push behavior**: `Eager` peers get the full bundle pushed immediately (today's `push_bundle` call, unchanged). `Lazy` peers get only the 40-byte `GossipIHave` digest. A peer receiving an `IHAVE` for a `(source, sequence)` it has NOT seen (checked against its own `seen_gossip_sequences` — the EXISTING `SeenSequenceState` structure, no new anti-replay state) issues a pull request (`GossipGraft{source_node_id, sequence}`) back to the digest-sender, which responds with the full signed bundle from its own `last_seen_bundles: HashMap<[u8;32], GossipBundle>` cache (bounded to one entry per known peer — same bound as today's `applied_endpoints` map).
  4. **Anti-forgery / no-downgrade-oracle property (the hard requirement)**: `GossipIHave`/`GossipGraft` are digest-and-pull-request messages ONLY — they carry no candidate data and are **never accepted as a substitute for signature verification**. An `IHAVE` claiming a sequence *lower* than what a peer already holds in `SeenSequenceState` is simply ignored (it cannot roll back the watermark — `seen_gossip_sequences` is monotonic-only and `IHAVE` never writes to it, only `ingest_inbound_bundle`'s existing accept path does). An `IHAVE` claiming a sequence *higher* than currently seen only ever triggers a `GRAFT` pull; the pulled bundle must independently pass the FULL existing `accept_bundle_with_now` pipeline (signature → freshness → strict-monotonic sequence → revoked-peer check) before it does anything. **No new trust decision is introduced**: the digest layer can only accelerate or suppress a push, never approve a bundle. A malicious peer sending false `IHAVE`s can at worst cause: (a) a spurious `GRAFT` pull (wasted bandwidth, bounded by rate-limiting the graft responder — new per-source token bucket, matching the existing relay rate-limiter pattern in `rustynet-relay`), or (b) a peer being wrongly demoted to `Lazy` for a source it actually needs (self-heals via the silence-window promotion in point 2 — worst case is a delayed propagation, not data loss, and the 30s heartbeat remint is an unconditional backstop that still exists unmodified).
  5. **Watermark/persistence interaction**: zero changes to `write_gossip_watermark`/`load_gossip_watermark` (gossip_runtime.rs:758-853) — the persist-before-commit invariant for `local_sequence` and the `seen` ledger is completely orthogonal to which wire message triggered the eventual `ingest_inbound_bundle` call. Digest/pull traffic never touches the watermark spool.
  6. **Bounded `seen_gossip_sequences` growth (a correctness fix riding along)**: add `unregister_peer(peer_id)` (currently absent — grounding confirms peers-map only grows) called from the revocation path, and prune `seen_gossip_sequences` / `PeerLinkMode` entries for any `node_id` no longer `Active` in the membership snapshot, driven off the SAME `revoked_peer_ids_from_membership` the code already calls at gossip_runtime.rs:543. This also fixes the sibling bug that `ordered_peer_ids_for_rebroadcast` never excludes `revoked_peer_ids`, so a revoked node currently still receives outbound pushes forever — with `PeerLinkMode` demotion/removal this collapses naturally: revoked peers get removed from the peer map entirely rather than merely reject-listed on ingest.

- **Why this is the intelligent version:** At 3 nodes this buys nothing measurable — 3×2=6 pushes per mint either way, and the digest/graft machinery adds pure overhead (more message types, more state) for zero benefit; Plumtree's own literature notes tree-based epidemic broadcast is a >dozens-of-nodes technique. At 50 nodes it is a real, honestly-quantifiable win: full flood is 50×49=2450 pushes of ≤705B per mint (≈1.7 MB mesh-wide per 30s cycle, ≈58 KB/s aggregate) plus a SECOND full re-flood wave per accepted bundle from every ingesting peer (so actual traffic is closer to O(N²) not just per-mint but per-accepted-bundle-hop — the current code has no hop limiting at all, meaning a single mint from node A generates a full-mesh re-push storm from every one of the other 49 nodes independently as each ingests and re-floods). With a converged Plumtree overlay, steady-state eager-tree fanout approaches O(N) total pushes (a spanning tree has N-1 edges) plus O(N) lazy digest hints (40 bytes each, ~2 KB/s at 50 nodes) — roughly a 20-40x reduction in bytes-on-wire at the top of the target range, which matters specifically for the Raspberry-Pi-class anchor (constrained uplink, likely also running relay/bundle-pull/enrollment-listener duties concurrently) and for occasionally-connected mobile clients where every wasted push is a battery/data cost.

- **Cost / tradeoffs:** New wire messages (`GossipIHave` 40B unsigned, `GossipGraft` 40B unsigned, `GossipGraftResponse` = existing signed bundle, no new format) inside the existing 4 KiB datagram cap and existing UDP port — no new listener, no new attack surface at the transport layer. New state: `PeerLinkMode` map (1 byte enum × N peers, negligible), `last_seen_bundle` cache (1 bundle × N peers, ≤705B×N ≈ 35 KB at 50 nodes — same order as today's `applied_endpoints`). New failure modes: (1) a partitioned lazy-link peer sees delayed (not lost) propagation bounded by the silence-window promotion + the unconditional 30s remint backstop, so worst-case staleness is one remint interval, not unbounded; (2) graft-request amplification if an attacker floods forged `IHAVE`s — mitigated by a per-source token bucket on the graft responder (new, small, mirrors `rustynet-relay`'s existing token-bucket pattern) and by `IHAVE`/`GRAFT` requiring the claimed source to be a `known_peer` (same `UnknownSource` check reused); (3) operational complexity increase — a new peer-link-mode state machine to reason about in incident debugging, and a new class of "why didn't I get this bundle promptly" question that requires inspecting lazy-link state, not just watermark state. Net new code is moderate (roughly the size of the existing `peer_gossip.rs` accept path again) — this is not a small patch.

- **Constraint check:**
  1. **No custom crypto / no VPN reinvention** — PASS. `GossipIHave`/`GossipGraft` are explicitly unsigned hint/pull messages that never gate a trust decision; the ONLY thing that ever mutates `seen_gossip_sequences` or `applied_endpoints` is the existing, unmodified `accept_bundle_with_now` → Ed25519 `verify_strict` pipeline. Digest traffic still rides inside the WireGuard tunnel (rustynet0) unchanged — no new crypto layer, no new transport.
  2. **Default-deny / fail-closed** — PASS. An `IHAVE` cannot downgrade a peer's sequence watermark (write-only via the accept path, never via digest). Missing/stale/ambiguous digest state (e.g. `PeerLinkMode` map not yet populated for a freshly-registered peer) defaults to `Eager` — i.e. defaults to the SAFE choice (full push, today's behavior), never to "assume the peer already has it and suppress delivery." A forged `GRAFT` at worst wastes a rate-limited response; it cannot manufacture acceptance of a bundle that fails signature/freshness/sequence checks.
  3. **Rust-first, no new runtime category** — PASS. Plumtree is a classical epidemic-protocol algorithm (control-theoretic gossip, not ML/inference); implementation is pure Rust additions to `gossip_runtime.rs`/`peer_gossip.rs`, no new dependency category.
  4. **Backend-abstraction boundary** — PASS (N/A). This lives entirely inside `rustynetd` (daemon-layer gossip runtime), touches no `rustynet-control`/`rustynet-policy`/`rustynet-dns-zone` domain crate, and introduces no WireGuard-specific type into any domain crate.
  5. **Resource profile (2-50 nodes, Pi-class anchor)** — PASS with the caveat stated in cost/tradeoffs: the win is real but modest in absolute terms at this scale (kilobytes/sec, not megabytes/sec) — this is a headroom/elegance improvement, not a fix for an active resource crisis on today's hardware, and the proposal says so plainly rather than oversell it.

- **Incremental build path:**
  - **Phase 1 (S, ≤half day):** Fix the two correctness bugs riding along regardless of the epidemic-fanout decision: add `unregister_peer`, exclude `revoked_peer_ids` from `ordered_peer_ids_for_rebroadcast`, and prune `seen_gossip_sequences` for non-Active nodes on membership refresh. Independently valuable, zero new wire messages, testable today against the existing `tests/gossip_three_peer_mesh.rs` harness.
  - **Phase 2 (M, 1-2 days):** Define `GossipIHave`/`GossipGraft` wire types + the `PeerLinkMode` map, defaulting every peer to `Eager` (behavior-identical to today). ## FIS-0004: Loss/RTT-variance congestion signal driving relay-preference + optional outbound pacing

- **Pipeline / area:** Dataplane (userspace-shared WireGuard backend pump) feeding the traversal/path-selection control plane (`rustynetd`).

- **Current behavior (grounded):** boringtun's `Tunn::stats()` (`third_party/boringtun/src/noise/mod.rs:573-585`) already computes, per peer, `(time_since_last_handshake, tx_bytes, rx_bytes, loss: f32, rtt: Option<u32>)`. `loss` is a real EWMA over the last `N_SESSIONS=8` WireGuard rekey sessions using each session's receive-counter validator (`expected` vs. `received`, weights 9.0/3.0/1.0/0.33/...), computed in `estimate_loss()` (`mod.rs:543-571`). `rtt` (`last_rtt: Option<u32>` ms) is set as a free side effect of the handshake round trip in `third_party/boringtun/src/noise/handshake.rs:646-647`. Both values are thrown away at the only two call sites in rustynet: `crates/rustynet-backend-wireguard/src/userspace_shared/engine.rs:341` — `let (_handshake, peer_tx, peer_rx, _loss, _rtt) = peer_state.tunnel.stats();` — and a second discard in the same file's test helper (~line 531). The cross-backend `TunnelStats` struct (`crates/rustynet-backend-api/src/lib.rs:103-110`) has no rtt/loss field and is engine-wide, not per-peer. The command-only backends (Linux `wg`-CLI, macOS wireguard-go-CLI, Windows WireGuardNT-CLI) never instantiate a boringtun `Tunn` in-process, so they have zero access to this signal — this proposal is userspace-shared-backend-only by construction. Path selection (`crates/rustynetd/src/traversal.rs::TraversalSession::on_direct_probe_timeout`, line ~1040) is driven purely by discrete events (N=3 consecutive raw UDP probe timeouts, `DEFAULT_TRAVERSAL_PROBE_RELAY_SWITCH_AFTER_FAILURES`) and coarse handshake-freshness (`crates/rustynetd/src/daemon.rs::traversal_handshake_is_fresh`, line 6037, 30s window) — there is no notion of "direct path is up but degrading."

- **Proposed mechanism:** Two independent, small, named classical estimators, computed inside the existing userspace pump tick (`userspace_shared/runtime.rs::run_worker`, the same loop that already calls `poll_authoritative_socket`/`poll_tun_device`/`expire_timed_out_round_trip`), with NO new packet formats and NO WireGuard/boringtun internals touched beyond reading the tuple boringtun already returns:
  1. **Loss-burst classifier** — un-discard boringtun's existing `loss: f32` per peer each tick and run it through a simple **AIMD-style hysteresis gate** (not a rate controller — just a debounced threshold, analogous to how TCP Reno treats 3 dup-acks as a congestion event rather than reacting to single-packet loss): flag `Degrading` only after the EWMA loss exceeds a threshold (e.g. 2%) for >=2 consecutive rekey-session windows (~4 minutes at default `REKEY_AFTER_TIME=120s`), clear back to `Healthy` after 1 clean window. This mirrors AIMD's "additive-increase-confidence, multiplicative-decrease-trust" asymmetry without implementing a full congestion window.
  2. **RTT-variance tracker** — maintain a smoothed RTT and RTT-variation per peer using the **TCP RFC 6298 SRTT/RTTVAR formulas** (`SRTT = SRTT + alpha*(RTT_sample - SRTT)`, `RTTVAR = RTTVAR + beta*(|SRTT-RTT_sample| - RTTVAR)`, alpha=1/8, beta=1/4 — the same constants Jacobson/Karels 1988 popularized and TCP still uses), fed by boringtun's `last_rtt` sample taken at each handshake/rekey (every <=120s) rather than per-packet (this is the honest scaling-down: no per-packet RTT sampling exists in WireGuard's protocol, so a **CoDel-style continuous delay signal is explicitly NOT available** — see cost section). A single new small struct, `PeerPathQuality { loss_state: LossState, srtt_ms: Option<u32>, rttvar_ms: Option<u32>, updated_at_unix: u64 }`, held per-peer in the engine (new `BTreeMap<NodeId, PeerPathQuality>` alongside the existing `peer_states`), is the only new data structure.
  - **Wire format:** none. This is a local-only derived signal, never serialized, never sent to a peer, never gossiped. It changes zero bytes on the wire.
  - **State machine:** `PeerPathQuality` becomes a **third input** (alongside existing handshake-freshness and probe-timeout-count) to `TraversalSession`'s reconsideration logic, surfaced through a new read-only accessor `UserspaceEngine::peer_path_quality(&self, node_id) -> Option<PeerPathQuality>` that the daemon polls once per traversal reconcile tick (already-existing ~1s cadence) — the same place `traversal_probe_due` already runs. It feeds exactly two consumers, both **advisory, never authoritative**:
    - (a) **Relay-preference input**: when `PathMode::Direct` and `loss_state == Degrading`, shorten the next-reprobe/re-race interval for a *fresh ICE pair race* (does not itself force a relay switch — that stays gated by the existing `execute_ice_pair_race` outcome) and set a `degrading_direct` flag surfaced in `rustynet status`/telemetry so the (separately-designed) path-migration policy can weight it. This proposal does not own "when to switch" — see sibling path-migration proposal for that decision surface; here it only makes the currently-blind "direct path is nominally up but rotten" state observable and cheaply actionable as an accelerant, never a forced cutover.
    - (b) **Optional outbound pacing for bulk flows**: a **LEDBAT-inspired (RFC 6817) yield-under-pressure gate**, not a bandwidth estimator — when `loss_state == Degrading` AND the per-peer TUN-egress queue depth in the pump tick exceeds a small watermark (bulk-flow heuristic: >8 packets queued for one peer in one 10ms tick, vs. the existing `MAX_TUN_PACKETS_PER_TICK=64` budget), the pump inserts a bounded artificial delay (max 2ms, capped, never blocking other peers) before dequeuing that peer's next batch — i.e., voluntarily *slow down bulk senders on a peer whose path is already degrading*, the same "yield to better-behaved traffic / don't add to the flood" philosophy LEDBAT uses for background transfers. This is pacing, not congestion control: no window, no ack-clocking, no rate estimate — just a defensive backoff so one degrading peer's bulk flow doesn't starve interactive traffic to the same peer sharing the single-threaded pump tick.

- **Why this is the intelligent version:** At 2-50 nodes, per-flow BBR-grade bandwidth-delay-product estimation is solving a problem Rustynet doesn't have — these are point-to-point WireGuard tunnels between a handful of home/mobile nodes, not a datacenter fabric with hundreds of competing flows needing fair-share bandwidth allocation. A full BBR model (RTT-min tracking, bottleneck-bandwidth probing cycles, pacing-gain cycling) needs continuous per-packet RTT/bandwidth samples that WireGuard's protocol does not produce (WireGuard has no per-packet ack; RTT is only observable at handshake/rekey granularity, every up to 120s) — implementing BBR here would mean *inventing* a synthetic packet-probing scheme on top of WireGuard just to feed the model, which both violates "no custom transport invention" in spirit and burns CPU/battery on a Pi-class anchor for a signal no one asked for. The honest scale-down is: **use the data boringtun already computes for free, at the granularity it's already sampled, and turn it from a discarded tuple into two named, bounded, classical estimators (AIMD-style hysteresis + RFC 6298 SRTT/RTTVAR)** — the same primitives TCP has used since 1988, sized to WireGuard's actual event rate (per-rekey, not per-packet). The win at 3-node scale is close to zero (there's rarely more than one candidate direct path and one relay to choose between, and a human watching `rustynet status` would notice degradation just as fast). The win at 50-node scale, multiple mobile/cellular clients, and an anchor serving as relay is real: today a direct path that has quietly degraded to 15% loss but hasn't fully died sits untouched until 3 consecutive raw probes time out or the handshake goes stale — this closes that blind spot cheaply, using data already being computed and thrown away.

- **Cost / tradeoffs:** **Wire bytes: zero** (no wire format changes; purely local telemetry). **CPU:** the loss/RTT computation is already happening inside boringtun's own `stats()` call (which is CHEAPER than the current pattern, since today rustynet calls `stats()` and discards two of five return values — this proposal captures values already paid for); the new hysteresis/SRTT math is O(1) per peer per stats-poll (bounded by however often the daemon polls `peer_path_quality`, proposed at the existing ~1s traversal-reconcile cadence, not the 10ms pump tick) — negligible on a Pi. **Memory:** one small fixed-size struct (`~32 bytes` with layout padding) per peer, bounded by mesh size (2-50 nodes), trivial. **New failure modes:** (1) a degrading-but-not-dead direct path could in principle cause reprobe-interval churn if the hysteresis thresholds are too twitchy — mitigated by the debounce (2 consecutive windows to flag, 1 clean window to clear) and by keeping this purely advisory to an already-existing, independently-gated ICE-race mechanism rather than a new forced-switch path; (2) the pacing gate (mechanism 2b) is the riskier piece — an artificial delay inserted into the single-threaded pump tick must be bounded and peer-scoped so it can never become a self-inflicted DoS on the whole node (cap at 2ms, only applies to the one degrading peer's queue, never touches the 10ms/64-packet budget invariants that already exist) — this needs careful benchmarking against the completed micro-perf pass to prove it doesn't regress the ~187ns/packet numbers for the common (non-degrading) case, which should be the overwhelming majority of ticks. **New attack surface:** minimal — this is a passive local reader of counters boringtun already trusts (WireGuard's own authenticated counter-validator feeds `estimate_loss`), not a new externally-influenced input; a malicious peer could induce apparent loss by dropping its own acks, but that only ever *biases toward preferring relay sooner*, which is the fail-safe direction (never grants capability, never bypasses default-deny), consistent with hard constraint 2. **Operational complexity:** two new tunable constants (loss threshold %, hysteresis window count) that need documenting alongside the existing `DEFAULT_TRAVERSAL_PROBE_*` family — risk of constant sprawl if not consolidated into the same config struct.

- **Constraint check:**
  1. *No custom crypto/VPN protocol invention* — **PASS**. Zero WireGuard protocol changes, zero new wire format, reads an existing boringtun return value verbatim. WireGuard stays fully behind `rustynet-backend-api`; this lives entirely inside `rustynet-backend-wireguard`'s userspace-shared adapter.
  2. *Default-deny / fail-closed, advisory-only decisions* — **PASS**. `PeerPathQuality` defaults to `Healthy`/`None` (no data = no opinion, never "assume degraded" or "assume healthy-enough-to-grant-anything"); absence or staleness (`updated_at_unix` too old) must resolve to "no signal" and defer entirely to the existing discrete state machine, never to a synthesized preference. It never grants capability or trust — it only shortens a reprobe timer or paces a queue, both already-safe operations on an already-authorized path.
  3. *Rust-first, no new runtime dependency category* — **PASS**. Pure arithmetic (EWMA, RFC 6298 formulas) in existing Rust, no new crate, no ML/embedded runtime.
  4. *Backend-abstraction boundary* — **PASS with a named seam**: `PeerPathQuality` lives inside `rustynet-backend-wireguard` (the adapter), and only a minimal advisory summary (e.g., a `bool degrading` or small enum) crosses into `rustynetd`'s traversal logic via a new trait method — this needs to be an abstract signal on `rustynet-backend-api::TunnelBackend`, NOT a boringtun-specific type leaking upward, matching the existing pattern of `peer_latest_handshake_unix`. `rustynet-control`/`rustynet-policy` are untouched entirely.
  5. *Pi-class / mobile / 2-50 node resource profile* — **PASS**. O(1) per-peer state, sub-microsecond arithmetic, no new allocation hot path, no per-packet overhead (samples at rekey/handshake cadence, ~120s, not per-packet).

- **Incremental build path:**
  - **Phase 1 (S, <=half day):** Un-discard `loss`/`rtt` in `engine.rs:stats()` and the test helper; add `PeerPathQuality` struct with the AIMD-hysteresis loss classifier only (no RTT yet); expose via a new per-peer diagnostic accessor; unit-test the hysteresis debounce logic in isolation (deterministic, no lab needed).
  - **Phase 2 (S-M, ~1 day):** Add the RFC 6298 SRTT/RTTVAR tracker fed by `last_rtt`; wire both into a small extension of the abstract `TunnelBackend` trait (mirroring `peer_latest_handshake_unix`'s pattern) so the daemon can read it without importing boringtun types; surface `degrading_direct` in `rustynet status` output as pure telemetry (no behavior change yet) — this alone is independently useful and low-risk to land/validate first.
  - **Phase 3 (M, ~1-2 days, needs the path-migration proposal's decision surface to exist first):** Wire the advisory flag into the reprobe-interval shortening for `PathMode::Direct` — smallest possible behavioral change (never forces relay, only accelerates the existing ICE-race reprobe cadence when the flag is set). Requires a design decision on where this input joins whatever weighting the path-migration proposal defines; this phase should not proceed until that decision surface is settled, to avoid two designs racing to define the same knob.
  - **Phase 4 (L, >=3 days, separately gated):** Outbound pacing gate (mechanism 2b) for bulk flows — highest-risk piece, needs its own perf-regression bench against the existing micro-perf baseline before it's allowed anywhere near the default-on path; should ship behind an explicit opt-in flag and be validated on the live-lab chaos/impairment profile (`profiles/live_lab/`) under induced loss before ever being considered for a default.
  - Each phase is independently revertable and none require touching boringtun's internals or the WireGuard wire protocol — the live-lab framework's existing chaos/impairment stages (`documents/operations/active/RustynetDataplaneExecutionPlan_2026-05-18.md` D5.1 NAT-profile lab, `symmetric`/impairment profiles) are a natural fit for inducing the loss/RTT-variance patterns Phase 1-3 need to be validated against, once such an impairment stage exists — none does yet for controlled loss/latency injection at the WireGuard-path level, so Phase 1-2 validation is unit/bench-only until that lab capability is separately built.

- **How you would know it worked / what would falsify it:** Success signal: in a live-lab run with induced asymmetric loss on one peer's direct path (once a loss-injection stage exists), the `degrading_direct` telemetry flag flips to true measurably before the existing 3-consecutive-probe-timeout relay fallback would have fired, and the accelerated reprobe cadence results in either (a) a healthier alternate ICE pair being found faster, or (b) a relay fallback recommendation reaching the operator/status surface earlier than today's blind 30s-freshness-window baseline — with zero false-positive flaps in a clean-path control run over the same duration. It would be **falsified** if: the hysteresis thresholds cause reprobe churn (repeated flag flap) on genuinely healthy paths under normal jitter (meaning the debounce window is wrong or WireGuard's per-rekey RTT sampling is simply too coarse-grained to be a useful continuous signal at all — in which case the honest conclusion is that RTT-variance tracking at 120s granularity doesn't carry enough information, and only the loss-burst half of the proposal survives); or if the pacing gate (Phase 4) is shown in bench to regress the common-case per-packet cost measurably against the completed micro-perf baseline, in which case Phase 4 should be dropped and only the measurement/advisory phases (1-3) retained.

- **Prior art:** Jacobson & Karels, "Congestion Avoidance and Control" (SIGCOMM 1988) — the AIMD and SRTT/RTTVAR smoothing this proposal directly reuses; RFC 6298 (Paxson/Allman/Chu/Chadha, TCP RTO computation) — exact SRTT/RTTVAR formulas cited; RFC 6817 (Shalunov et al., LEDBAT) — the "yield under pressure, don't add to the flood" philosophy behind the optional pacing gate; RFC 8985 / Cardwell et al., "BBR: Congestion-Based Congestion Control" (ACM Queue 2016) — cited specifically to justify NOT adopting it (needs continuous per-packet bandwidth-delay-product sampling WireGuard's protocol doesn't provide); Nichols & Jacobson, "Controlling Queue Delay" (CoDel, ACM Queue 2012) — cited to note the continuous sojourn-time delay signal CoDel needs is likewise unavailable at WireGuard's per-rekey sampling granularity, so this proposal deliberately does not claim CoDel-equivalence, only CoDel-inspired directionality (react to delay/loss trend, not instantaneous single-sample noise).

- **Adversarial verification (independent citation fact-check + constraint review, 2026-07-01):** Citation fact-check: **1 correction applied** in the text above (struct estimate revised ~24 → ~32 bytes with layout padding). Constraint review verdict: **accept-with-fixes** —
  - Pin the backend seam before Phase 2: the concrete `PeerPathQuality` struct (LossState/SRTT/RTTVAR naming) stays inside `rustynet-backend-wireguard`; only a small primitive signal (a bool or 3-state `PathHealth` enum) crosses the `rustynet-backend-api` trait boundary, mirroring the existing `peer_latest_handshake_unix` pattern (`lib.rs:184`) — otherwise boringtun-shaped concepts silently leak into `rustynetd`.
  - The pacing gate (mechanism 2b) must respect the actual concurrency model: `run_worker` (`crates/rustynet-backend-wireguard/src/userspace_shared/runtime.rs:902-990`) is ONE synchronous OS thread per engine serving ALL peers per tick — any per-peer pacing delay must be non-blocking (skip-and-revisit scheduling), never a sleep on the shared worker thread, or one degrading peer stalls every peer.
  - Before Phase 3, verify accelerated reprobing stays inside the existing probe rate limits: a peer inducing apparent loss to amplify reprobe frequency is a plausible (low-severity, self-limited) nudge vector the current text only handles for the relay-preference direction.
  - The live-lab loss-injection stage is a **blocking prerequisite for Phase 3 sign-off**, not merely a Phase 1-2 validation nicety — listed explicitly so it cannot be silently skipped.
- **Designer's own weakest point (disclosed):** the RTT-variance relay-preference trigger adds a second path-quality signal alongside the existing discrete handshake-freshness/probe-timeout state machine; the two must be proven non-oscillating (see FIS-0001's hysteresis discipline, which this proposal must share constants with) before this is more than a diagnostic overlay.

## FIS-0005: Decision-support scoring engine for anchor/relay/exit role placement
- **Pipeline / area:** Admin/operator UX + a new optional telemetry-aggregation slice sitting beside (not inside) `rustynet-control` membership/gossip. Touches `rustynet-cli` (new `rustynet role recommend` surface), `rustynet-operator` (menu item), and — only in the latest phase — an additive gossip/telemetry channel read by `rustynetd`.

- **Current behavior (grounded):** Role assignment is entirely manual and blind to runtime signal. `rustynet role set <preset>` parses to `CliCommand::Role` and runs through `execute_role_plan` (`crates/rustynet-cli/src/main.rs:17349`), which executes `plan_concrete_actions` (`crates/rustynet-cli/src/role_cli.rs:468`) — a pure function of `(current, target)` RolePreset pairs with **zero fitness input**. `transition_plan` (`crates/rustynet-control/src/role_presets.rs:519`) classifies every transition as `Identity | LocalOnly | Blocked | SignedMembership | Irreversible` (`TransitionKind`, line 429) and `requires_owner_signature()` (line 461) gates capability-changing transitions on operator signing — there is no code path today that could bypass this even if it wanted to. `MembershipNode` (`crates/rustynet-control/src/membership.rs:139`) has exactly 8 fields (node_id, node_pubkey_hex, owner, status, roles, capabilities, joined_at_unix, updated_at_unix) — no uptime/NAT/bandwidth/latency field exists in signed state. `GossipBundle` (`crates/rustynetd/src/peer_gossip.rs:106`) carries only `CandidateSet` endpoint data — no telemetry. Each daemon *does* hold rich live telemetry, but it is exposed only as a single space-separated key=value IPC status line for local queries (`crates/rustynetd/src/daemon.rs:7148`, the giant `format!` at that line: `path_live_direct_peers`, `traversal_probe_result`, `membership_epoch`, etc.) — nothing aggregates this across nodes. `select_port_mapping_authority_node_id` (`crates/rustynetd/src/gossip_runtime.rs:560`) and `anchor_gossip_seed_peer_ids_from_membership` (line 524) are the one existing "placement" mechanism in the mesh, and both are static, capability-derived, zero-telemetry elections (lexicographically-smallest node_id, or membership-capability-tagged set). NAT classification exists only as an on-demand CLI-local heuristic: `classify_nat` (`crates/rustynet-cli/src/ops_cross_network_preflight.rs:355`) → `NatClassHeuristic` (line 127: ConeNatLikely/PortRestrictedLikely/SymmetricLikely/InsufficientData), printed/JSON'd by `run_cross_network_preflight` (line 183) and never persisted or shared. Read-only hooks an advisory layer could plug into today, verified: `NodeCommand::Info`/`List` (`main.rs:18677`, `18708`) reading the local membership snapshot only; `RoleCommand::TransitionCheck` (`main.rs:17304`, parsed at line 1545-1548) — a pure "what would happen" preview; and the operator `MenuTree` (`crates/rustynet-operator/src/menu.rs:51`) which already lists `ShowRoleStatus`/`Netcheck`/`Status`/`ListRoles` as read-only actions alongside the mutating ones.

- **Proposed mechanism:** A **weighted multi-criteria decision analysis (MCDA) scorer** (Keeney & Raiffa's additive value model, the standard small-N technique — not AHP, which needs pairwise-comparison overhead unjustified at this scale) combined with **exact betweenness/closeness centrality** (Brandes' algorithm, O(V·E) — trivial at V≤50) computed over the mesh's *observed* connectivity graph, and a **Raft-inspired leadership-transfer desirability check** (borrowing Raft's leadership-transfer precondition — "only hand off to a follower whose log is at least as up to date" — reinterpreted as "only recommend anchor succession to a candidate whose observed health strictly dominates the incumbent's, sustained over a decay window" so recommendations aren't noise-triggered by one bad sample).
  Concrete Rustynet adaptation, phased (see build path):
  1. A **new read-only local aggregator crate `rustynet-advisor`** (sits beside `rustynet-cli`/`rustynet-operator`, depends only on `rustynet-control`'s abstract types — respects §8/10.3, never touches backend types) collects per-node observations. Phase 1-2 source: SSH/loopback fan-out of the existing IPC `status` line across nodes the admin already has access to (same trust boundary the live-lab orchestrator uses today, out-of-band) — no wire change. Phase 3 (gated, see tradeoffs) source: a *new, additive* gossip message type, `TelemetryDigest` — NOT a mutation of `GossipBundle` v1 (whose 705-byte fixed layout at `peer_gossip.rs:73-80` is pinned by test) but a second bundle type sharing the same Ed25519 signature + sequence + freshness-window discipline, bumping the gossip demux (not `GOSSIP_BUNDLE_WIRE_VERSION`) to dispatch on a type byte. Payload: 5 scalar fields only — `uptime_ratio_ewma` (u16 fixed-point), `nat_class_code` (u8, reusing the existing `NatClassHeuristic` 4-value enum, computed locally via the *existing* `classify_nat` running on a schedule instead of on-demand), `handshake_success_ewma` (u16), `bandwidth_headroom_class` (u8, coarse 4-bucket: unknown/low/medium/high — never a raw byte counter, to avoid becoming a bandwidth-measurement feature creeping into privacy/attack-surface territory), `observed_at_unix`. No new signed *capability* — this is data, never trust.
  2. **MCDA scorer** (Phase 1, local to `rustynet-advisor`, pure function `score_candidate(criteria) -> f64`): fixed weights `w = {uptime: 0.30, reachability_stability: 0.25, nat_class: 0.20, bandwidth_headroom: 0.15, centrality: 0.10}`, each criterion normalized to [0,1], additive value `Σ wᵢ·vᵢ`. Weights are a config-file constant (`documents/operations/active` or a `--weights` flag), not learned — no ML runtime (§3 constraint), just documented judgment calls: uptime and reachability dominate because a flapping anchor is worse than a slow one; NAT class matters because `symmetric_likely` disqualifies relay/exit-facing roles per the existing `PreflightVerdict::RelayRequired` semantics; centrality is weighted lowest because in a 2-50 node home mesh with full-mesh WireGuard peering, betweenness rarely differentiates candidates (see honesty note below).
  3. **Betweenness centrality** (Phase 2, `M`): build the mesh graph from `MembershipState.nodes` (structural — every active node is a WG peer of every other by current architecture) weighted by *live* edge health from aggregated status (`path_live_direct_peers` vs relay fallback per edge). Brandes' algorithm, O(V·E) ≈ O(50·2500) worst case — microseconds. This *only* differentiates candidates when the mesh isn't fully connected (some peers relay-only) — flagged honestly below as usually near-uniform in this topology.
  4. **Degradation/succession heuristic** (Phase 2-3): incumbent anchor's EWMA health score is compared each evaluation tick (piggybacking the existing 1000ms `DEFAULT_RECONCILE_INTERVAL_MS` cadence conceptually, but run out-of-band in `rustynet-advisor`, not in the daemon hot path) against the best challenger's score with a **hysteresis band + minimum-dwell-time** (Raft leadership-transfer style: don't recommend transfer on a single bad sample; require the challenger to dominate for N consecutive windows, e.g. 3×30s = 90s, mirroring the existing `GOSSIP_REMINT_INTERVAL_SECS=30` cadence already in the codebase) before emitting a `RoleRecommendation`.
  5. **Surfacing** (Phase 1, `S`): `rustynet role recommend [--role anchor|relay|exit] [--json]` — a strictly read-only CLI verb parallel to the already-existing `role transition-check --to` (`main.rs:17304`) — prints a ranked candidate list + score breakdown + rationale string, nothing more. Operator menu gets one new read-only `MenuAction::ShowRoleRecommendation` item alongside the existing `ShowRoleStatus`/`Netcheck` (`menu.rs:51-79`). The output of `role recommend` is deliberately structured to be paste-ready into the *existing* `rustynet anchor advertise` unsigned-proposal flow (`main.rs:6248`, which already builds a `MembershipUpdateRecord` for out-of-band signing) — the admin still runs the real signing command themselves; the advisor never invokes it.

- **Why this is the intelligent version:** At 2-3 nodes this is close to useless — there's often exactly one plausible anchor (the always-on home server) and the admin already knows it; the "intelligence" buys almost nothing over "look at `node list`". The honest win shows up at 15-50 nodes with multiple always-on candidates (a home server, a NAS box, a spare laptop-turned-anchor, several phones): today the admin has literally no aggregated view and must SSH into each box and read `status` by hand, or trust vibes. A weighted score with an explained breakdown ("candidate B: uptime 0.95, but symmetric NAT — excluded from relay candidacy") converts an O(N) manual telemetry-reading chore into a single ranked list, and the degradation heuristic (Raft-style dwell/hysteresis) is the part that actually earns "intelligent" — it's the difference between "the anchor's WG handshake glitched once" and "the anchor has been meaningfully worse than an alternative for 90 seconds straight," which prevents flapping *recommendations* (not flapping capability — capability never auto-changes here) the way naive threshold-crossing would.

- **Cost / tradeoffs:** Phase 1-2 (SSH/status-line fan-out, local scoring, CLI/menu surfacing): near-zero — a few KB of Rust, no wire change, no new attack surface beyond what admin SSH access already implies, negligible CPU (Brandes on V≤50 is sub-millisecond). Phase 3 (gossiped `TelemetryDigest`) is where real cost appears: a new signed message type is new attack surface (another thing to replay-window/freshness-check/signature-verify, mirroring the existing `peer_gossip.rs` discipline but doubling the reviewed surface), extra wire bytes (roughly the same ~85-byte order of magnitude as a `GossipBundle`, sent on the same 30s heartbeat — negligible at N≤50, but it's O(N) new bytes/sec of *signed* traffic that didn't exist before), and — most importantly — it creates a **new signal an attacker with one compromised low-trust node could try to game** (report inflated fake uptime/NAT-class to bias the *admin's attention*, not their capability, toward that node — mitigated by the hard constraint that this can never auto-grant, only bias what a human looks at first, but "the admin's attention" is itself a security-relevant resource worth naming honestly). Operational complexity: one more crate, one more config surface (weights), one more thing that can be silently stale (see fail-closed section below) and confuse an admin who doesn't read the "insufficient data" caveat.

- **Constraint check:**
  1. *No custom crypto/VPN protocol* — PASS. `TelemetryDigest` reuses the exact Ed25519-sign-and-verify discipline already implemented for `GossipBundle` (`peer_gossip.rs` domain separation + freshness window pattern); WireGuard is untouched, telemetry rides *inside* the existing tunnel exactly as gossip already does (`gossip_transport.rs`, port 51821).
  2. *Default-deny / fail-closed, never silently grants capability* — PASS by construction: `rustynet-advisor` has no write path into `rustynet-control` membership, no call to `execute_role_plan`, no signing key access. Missing/stale/ambiguous telemetry (no observation within the freshness window, `InsufficientData` NAT class, fewer than 2 candidates) must yield an explicit `NoRecommendation { reason }` rather than a best-effort guess — this is the one place the design must be reviewed hardest, since a scoring engine's natural failure mode is "confidently rank garbage," which the fail-closed rule explicitly forbids. Elevated capability changes still require the existing `requires_owner_signature()` (`role_presets.rs:461`) signed flow; the advisor emits text, a human runs `role set`/`anchor advertise` themselves.
  3. *Rust-first, no new runtime category* — PASS. MCDA is a weighted sum; Brandes' betweenness is a textbook graph algorithm (BFS/Dijkstra-based); the succession heuristic is a hysteresis/dwell-time state machine. All are plain Rust, no Python/ONNX/inference-service dependency, matching the "classical algorithms are the intelligence bar" instruction directly.
  4. *Backend-abstraction boundary* — PASS. `rustynet-advisor` reads `MembershipState`/`RoleCapability` (already transport-agnostic types from `rustynet-control`) and the IPC status line (already a stable text protocol); it never imports `rustynet-backend-wireguard` or boringtun types.
  5. *Resource profile (Pi-class anchor, 2-50 nodes)* — PASS for Phases 1-2 (this runs on the *admin's* laptop/CLI, not on the Pi anchor itself). Phase 3's gossip addition runs on every node including the Pi anchor, but the added CPU/memory is the same order of magnitude as the existing gossip mint/verify path already running there today, and V≤50 keeps Brandes trivial — the design explicitly rejects any per-node telemetry history/time-series store (a datacenter-monitoring pattern) in favor of a single EWMA scalar per criterion, precisely to stay off the Pi's limited flash-write budget.

- **Incremental build path:**
  - Phase 1 (S, <half day): `rustynet-advisor` crate skeleton + MCDA scorer (`score_candidate`) as a pure unit-tested function over a manually-constructed `Vec<CandidateObservation>` struct; no telemetry sourcing yet, just the math + weight config + the "insufficient data → no recommendation" fail-closed path as the first test (mirrors the repo convention "first test is empty input → deny").
  - Phase 2 (M, 1-2 days): SSH/local-status fan-out collector (reuses the live-lab orchestrator's existing SSH-to-node pattern, out-of-band, no new trust surface) parsing the *existing* IPC status line into `CandidateObservation`; wire up `rustynet role recommend` CLI verb + operator menu item; add Brandes' betweenness over the membership-derived graph. This phase alone is independently shippable and already delivers the core "admin-side aggregation" gap called out in the grounding notes, with zero wire/protocol change — a live-lab stage could validate it by standing up N nodes, deliberately degrading one (kill its WG interface), and asserting `role recommend --role anchor` demotes it.
  - Phase 3 (L, needs a design decision — flag for explicit sign-off before building, same posture as the D14.e/f gates in the dataplane plan): the gossiped `TelemetryDigest` wire type, so recommendations work without SSH access to every node (e.g. recommending a mobile client for a role while the admin isn't at their desk). This is the piece that should get an actual security review pass (replay window, signature domain separation, demux versioning) before landing, not bundled into Phase 1-2's shipping.

## FIS-0006: SPRT/CUSUM Flake-vs-Regression Classifier over the Live-Lab Run Matrix
- **Pipeline / area:** Live-lab evidence pipeline — run-matrix analytics and orchestrator re-run scheduling (`documents/operations/live_lab_run_matrix.csv`, `crates/rustynet-cli/src/vm_lab/overnight/backlog.rs`, `crates/rustynet-cli/src/vm_lab/orchestrator/runner.rs`).

- **Current behavior (grounded):** Three places touch "is this stage flaky or really broken?" today, and none of them compute it from history:
  1. `backlog.rs` `CellState` (L159-184) has a `Flaky` variant with `base_value(self) -> u32` (L207-218) giving it priority 50 (between `Red`=70 and `Unknown`=30). `PriorVerdict` (L221-231) has a `Flaky` variant, and `PriorVerdict::from_status` (L233-241) maps the literal token `"flaky"` from an operator-typed `--seed-status os:role=status` string (see `parse_seed_status`, ~L246+). Grep confirms no function anywhere derives `Flaky` from run history — it is purely human-asserted.
  2. `crates/rustynet-mcp/src/bin/lab_state.rs` already ships `get_run_trend` (tool registered ~L2795, dispatched ~L3175, body L967+) which reads the last N rows of the *whole* `live_lab_run_matrix.csv` and calls `trend_verdict()` (L2171-2215): a deterministic streak-count heuristic — "GREEN stable" if last 2 rows pass, `"STUCK at <stage>"` if the last ≥2 failing rows share `first_failed_stage`, `"MOVING"` if the failing stage keeps changing. This is a naive baseline with no probability model, no keying by topology, and no error-rate guarantee (a streak of 2 identical failures is called "STUCK" regardless of whether that stage fails 5% or 95% of the time in general).
  3. `runner.rs` `StateMachineRunner::run` (L33-73) and `topological_order` (L79-124) are stateless per invocation: dependency order + skip-cascade only, no read of prior runs, no re-run prioritization signal in or out.
  The matrix itself (414 rows) already contains the raw material for a real per-(stage, topology) series: e.g. `phase24_macos_smoke_retry36/38/39/40/41/42/43/44` (8 runs, distinct commits `66184f4`/`2746333`(×2)/`a1c064f`/`27b0e39`/`56b1776`/`abd8ad8`/`1c254ff`, `first_failed_stage` alternating between `live_anchor`, `bootstrap_hosts`, `vm_lab_setup_live_lab`, `cleanup_hosts`) and the `rn_live_3node_r*` sequence (16 distinct runs under one `report_dir` topology prefix). `topology_summary` and `profile_path` columns are already present per row and are usable topology-signature keys without a schema change.
  The `OvernightAutonomousBugHuntProposal_2026-06-08.md` explicitly disclaims this: §2 "Non-Goals" states "Not a flake-hunter... a separate mode tracked under §12 future work"; §16 "Out of Scope (Future)" lists "Soak/flake mode" and "Multi-night trend analysis — track per-cell pass-rate and progress velocity across nights" as unbuilt future work. This candidate is that deferred work, scoped as R&D.

- **Proposed mechanism:** Wald's Sequential Probability Ratio Test (SPRT, Wald 1945) and Page's CUSUM control chart (Page 1954), applied per `(stage_id, topology_signature)` outcome series, Bernoulli-coded (pass=0, fail=1), ordered by the matrix's existing chronological row order (a commit-ordering proxy — the CSV is append-only in run order, not sorted by `git_commit` topology, so "commit boundary" below means *row-order* boundary unless a future phase adds real commit-graph ordering):
  - **CUSUM** detects a shift in failure probability from a baseline `p0` (the historical/nominal pass rate for that cell) to an elevated `p1`. Two one-sided cumulative sums:
    `S+_n = max(0, S+_{n-1} + (x_n - k))`, `S-_n = min(0, S-_{n-1} + (x_n + k))`
    where `x_n` is the centered observation (e.g. `x_n = 1{fail} - p0`), and `k` is a reference/slack value (typically half the shift-to-detect, `k = (p1 - p0)/2`). A signal fires when `S+_n > h` (or `S-_n < -h`) for decision interval `h`, chosen from a target in-control ARL0 (average run length before a false alarm) via standard CUSUM ARL tables/Siegmund's approximation, not a magic number.
  - **SPRT** frames it as sequential hypothesis testing: H0 "stationary flaky, p = p0" vs H1 "regressed, p = p1 > p0". The log-likelihood ratio after n observations,
    `Λ_n = Σ log( p1^{x_i} (1-p1)^{1-x_i} / p0^{x_i} (1-p0)^{1-x_i} )`,
    is compared against thresholds `A = log((1-β)/α)` and `B = log(β/(1-α))` derived from target false-alarm rate `α` and miss rate `β` (Wald's formulas). Accept H1 (real regression, escalate) when `Λ_n ≥ A`; accept H0 (flaky/no shift) when `Λ_n ≤ B`; otherwise continue sampling (this is SPRT's core property — it doesn't force a verdict on insufficient evidence, which is exactly the honest behavior wanted here).
  - Output is a **label + confidence class**, not a boolean: `{FlakyStationary, RegressionSuspected, Indeterminate(pending N more runs)}`, always accompanied by the observation count so a 3-observation "RegressionSuspected" cannot be mistaken for a 40-observation one.

- **Why this is the intelligent version:** `trend_verdict()`'s "2 in a row = STUCK" rule has no error-rate semantics — it flags exactly as confidently on a cell with true `p=0.15` (occasional flake, 2-in-a-row is a ~2% event, arguably meaningful) as one with true `p=0.6` (frequent flake, 2-in-a-row is a ~36% event, not meaningful) because it never estimates `p` at all. SPRT/CUSUM give a **calibrated** decision: the same "2 consecutive same-stage fails" evidence produces a different verdict depending on the cell's actual historical variance, because the thresholds are derived from an explicit target false-positive/false-negative rate rather than a fixed streak count. This turns "is this stage flaky" from an eyeballed guess into a testable statistical claim with a stated error budget — the same evidentiary standard the repo already applies to code (gates, not vibes) applied to its own test-signal quality.

- **Cost / tradeoffs:** Be honest: this is the proposal's weakest point and must be stated plainly. The matrix has 414 rows total across dozens of distinct `(stage_id, topology_signature)` pairs; per-cell sample sizes will mostly be single digits (the two example series found have 8 and 16 observations — among the *longest* runs in the whole matrix). SPRT and CUSUM are asymptotic/large-sample-friendly tools; at n<10 the log-likelihood ratio is dominated by whichever few observations happened to occur, and a naive `p0` estimated from the same short series is circular (using data to set the null it's tested against). Consequences: (a) most cells will sit in `Indeterminate` — which is the CORRECT behavior for SPRT (it is designed to say "not enough evidence yet," unlike a heuristic that always emits a verdict), but it means the tool will be silent on the majority of the matrix for a long time; (b) `p0`/`p1` cannot be tuned per cell from local history alone at these volumes — the practical fallback is a small number of *pooled* priors (e.g. "setup-class stages," "role-activation-class stages," "network-flap-class stages") derived from the whole matrix, explicitly documented as a shared prior, not per-cell fitting; (c) CUSUM's `h`/`k` and SPRT's `A`/`B` require an explicit, documented `α`/`β` choice — under-specifying them (or picking them to make the demo look good) is the single most likely way this gets misused, so the constant needs to live in one reviewed constants module with a comment citing the derivation, not be hand-tuned per stage. Engineering cost is small (pure arithmetic over an already-parsed CSV structure, no new dependency), but the *statistical validity* cost is real and must be disclosed in the tool's own output, not just this doc.

- **Constraint check:**
  1. No cryptography/VPN protocol touched — this is CSV analytics tooling in `rustynet-cli`/`rustynet-lab-monitor`, nowhere near `rustynet-backend-*`. Pass.
  2. Default-deny / fail-closed: the classifier is strictly advisory. It may set `CellState::Flaky`'s *priority weight* or annotate a `RunTrendReport` with a label; it MUST NOT flip `overall_result`, MUST NOT mark a `fail` row as passing evidence, and MUST NOT feed into any gate or the live-lab-run-matrix append path (`live_lab_run_matrix.rs::append_live_lab_run_matrix_row`) as a substitute for a real run. On missing/ambiguous data (n too small, `p0` unknown) it defaults to `Indeterminate` (no claim), never to a green/flaky label by default. This directly matches the FAIL-LOUD / no-dry-run-as-pass culture — it changes scheduling priority only, never evidence. Pass.
  3. Rust-first, no new dependency category: CUSUM/SPRT are ~50 lines of arithmetic over `f64`/`u32` counters (running sum, log-ratio accumulation) — no stats crate needed, matching the "classical algorithms are the intelligence bar" instruction. Pass.
  4. Domain crates untouched — this lives entirely in `rustynet-cli` (orchestrator/overnight) and optionally `rustynet-lab-monitor`; `rustynet-control`/`rustynet-policy`/`rustynet-dns-zone` are not touched. Pass.
  5. Raspberry-Pi/laptop/2-50-node scale, Mac-UTM lab host: computation is O(rows) over a 414-row CSV (microseconds), run on the operator's Mac, not the lab nodes. No datacenter mechanism (no distributed state store, no streaming pipeline) — it's a batch pass over one CSV file. Pass.

  - **Phase 0 (S):** Add a `run_history.rs` module (or extend `live_lab_run_matrix.rs`) that groups existing matrix rows into `(stage_id, topology_signature)` series and reproduces `trend_verdict`'s existing streak logic as the documented baseline/null-model — no new statistics yet, just the grouping data structure. This makes the *comparison* honest and testable before adding the new math.

  - **Phase 3 (L, needs a design decision):** Decide and document the pooled-prior policy for small-n cells (stage-class buckets vs. per-cell-only-when-n≥threshold) — this is a judgment call about acceptable false-positive rate at this evidence volume and should not be made silently inside code.

- **How you would know it worked / what would falsify it:** Retrospective backtest against the existing 414-row matrix is the falsification test: extract every `(stage_id, topology_signature)` series with ≥6 observations (a handful exist today, e.g. the two identified sequences), replay the CUSUM/SPRT classifier row-by-row in original order, and compare its verdicts against the human-written `notes`/`regression_notes` free-text in those same rows (which often already state "different failure mode" or "same failure mode as X"). ## FIS-0007: Load-aware relay selection + per-session DRR fairness on the relay

- **Pipeline / area:** `crates/rustynet-relay` (dataplane forwarding + admission), `crates/rustynetd/src/daemon.rs` (relay candidate selection), `crates/rustynet-control` (signed relay-fleet bundle) — cross-network dataplane track (D2-D14 lineage), specifically the currently-unwired `RelayFleet` selection code and the zero-fairness forwarding loop.

- **Current behavior (grounded):**
  - Selection: `select_runtime_relay_candidate_with_verified_fleet` (`crates/rustynetd/src/daemon.rs:13689`) takes `max_by_key(|c| c.priority)` over the peer's signed traversal-bundle relay candidates — a static, peer-authored priority, tie order unspecified — then only *validates* the winner against the signed `SignedRelayFleetBundle` (relay_id + endpoint membership, `ensure_runtime_relay_candidate_in_verified_fleet`, line 13716). `RelayFleetNodeDescriptor.priority`/`.capacity` (`crates/rustynet-control/src/lib.rs:1442`) are parsed and validated (`capacity>0` only, `validate_runtime_relay_fleet_descriptor`) but never read for choice. Separately, `RelayFleet::select_with_policy` (`crates/rustynet-relay/src/lib.rs:43-76`) already implements healthy-filter + region-policy + latency-sort selection with `mark_unhealthy` failover — but `rg` across the workspace shows its only callers are its own unit tests (`relay_fleet_prefers_healthy_low_latency_nodes` and siblings, `lib.rs` bottom). No production code path constructs a `RelayFleet` or feeds it real latency.
  - Forwarding: verified `spawn_forward_task` (`crates/rustynet-relay/src/main.rs:530-604`) — one tokio task per allocated port, `try_recv_from` into a stack buffer, on WouldBlock a bare `tokio::time::sleep(Duration::from_micros(100))`, on a frame a `transport.write().await` (single `Arc<RwLock<RelayTransport>>` shared by every session on the process) then a zero-copy `send_to` whose result is discarded (`let _ = peer_socket.send_to(...)`). There is no queue, no per-session byte/packet accounting beyond the existing per-node_id token bucket (`crates/rustynet-relay/src/rate_limit.rs`), and no scheduling discipline between sessions sharing the write lock — order of service is whichever task's `.await` wakes and acquires the lock first.
  - Signal starvation confirmed: the relay's only externally-visible load signal is the loopback-only `/healthz`/`/metrics` (`HealthSnapshot`, `crates/rustynet-relay/src/main.rs:904`) with exactly 4 gauges (active_sessions, allocated_ports, max_sessions_per_node, max_total_sessions) — not reachable by peers, and no drop/rate-limit-hit/throughput counters exist even locally.
  - Reject-reason confirmed collapsed on the wire: `RejectReason` (`crates/rustynet-relay/src/transport.rs:69`) is a real Rust enum with distinct variants including `Capacity` (session-cap exceeded, lines 417/432), but `serialize_relay_reject()` (`crates/rustynet-relay/src/main.rs:840`) always emits the fixed wire constant `RELAY_REJECT_GENERIC_REASON = "Rejected"` (line 233) — the client cannot distinguish "you're rate-limited/full" from "your token is bad," so no client can react differently to a busy relay today.

- **Proposed mechanism:**
  (a) **Power-of-two-choices relay selection** (Mitzenmacher 2001, "The Power of Two Choices in Randomized Load Balancing") applied when a peer's signed traversal bundle carries ≥2 relay candidates: instead of `max_by_key(priority)`, the client samples 2 of the N relay candidates (deterministic pick via a session-scoped RNG seed to stay reproducible for tests), asks each a cheap load question, and picks the lower-loaded of the two. The load signal itself must NOT be a new authenticated per-packet mechanism (PlugAndPlayPlan §9.4 explicitly rejects that for v1 and I am not arguing against it — this proposal keeps that decision). Instead the signal rides two channels that already exist: (1) the existing signed `RelayHelloAck` gets one new field, `load_hint: u8` (0-255, coarse bucketed utilization = `min(255, active_sessions * 255 / max_total_sessions)`), computed relay-side at ack time and returned over the *already-established, already-authenticated* hello/ack exchange — no new wire message, no new trust surface, because the ack is already relay-to-this-client-only and unforgeable-by-others (UDP source-tuple bound). (2) For pre-connection candidate ranking (before any hello is sent), the relay operator publishes an updated `capacity`/current-load estimate into the *existing* signed `RelayFleetBundle` refresh cycle (already re-signed and redistributed by the fleet operator on its existing cadence, ≤300s TTL) — reusing the already-signed, already-validated `RelayFleetNodeDescriptor.capacity` field that today is parsed but discarded. Nothing new is trusted: the load hint from (1) only ever picks between two relays *already* individually verified via `ensure_runtime_relay_candidate_in_verified_fleet`, and a missing/stale/zero load_hint (e.g. relay didn't set it, or bundle past TTL) makes the algorithm fall back to the existing `max_by_key(priority)` behavior — fail closed to today's deterministic choice, never fail open to an unverified relay. `RelayFleet::select_with_policy` gets wired as the actual implementation of the "ask 2, pick lower" comparator, finally giving it a caller.
  (b) **Deficit Round Robin** (Shreedhar & Varghese, 1996, "Efficient Fair Queueing Using Deficit Round Robin") inside one relay's forwarding path, replacing the current lock-order-is-scheduling-order behavior. Concretely: each `RelaySession` gains a `deficit_counter: u32` and forwarding moves from "recv → immediately send" per port-task to "recv → enqueue in a small bounded per-session ring buffer (e.g. 32 frames) → a single scheduler task drains sessions in DRR order (quantum = 1500 bytes, one MTU-sized frame per credited round) across all sessions sharing a destination socket-write path." This targets a *hypothesized* fairness gap (to be confirmed by the lock-wait-time trace in the falsification plan before Phase 4 is approved): the `transport.write()` lock is held only for the brief in-memory `forward_packet` bookkeeping — the `send_to` runs under a separate read lock on `allocated_sockets` — so raw lock contention is likely mild; the more plausible unfairness source is the flat 100µs poll-sleep floor per port task plus tokio's scheduler deciding wake order, under which a bulk-transfer session can still crowd out a latency-sensitive one because there is no scheduling discipline at all between sessions. DRR bounds this: no session can consume more than its fair share of forwarding rounds regardless of arrival rate. This does not touch the WireGuard tunnel or its crypto — DRR operates purely on already-decrypted-adjacent ciphertext UDP frames the relay forwards blindly; the relay still never decrypts payload.

- **Why this is the intelligent version:** Part (a) only has a decision to make once ≥2 relays exist on a mesh — at exactly 1 relay (today's only lab-proven configuration, see D11/D5.1 "live pass criterion still open" status) the power-of-two comparator has nothing to choose between and degenerates to a no-op identical to current behavior; the win is real only at the "several relays, 10-50 node mesh" scale the parity docs gesture toward but haven't built yet. Part (b) is a genuine improvement even at today's proven scale (a single Raspberry-Pi relay, 2-8 sessions per node, up to 4096 total sessions): the single global write lock is confirmed serializing ALL forwarding, so at even modest concurrent session counts (a laptop doing a large file sync + a phone on a voice call, both relayed) the current design has no mechanism to stop the file sync from monopolizing forwarding slots. DRR is the textbook fix at the exact scale this project targets (it was designed for exactly this: bounded memory, O(1) per-packet work, no floating point, cheap enough for a Pi).

- **Cost / tradeoffs:**
  - Wire: +1 byte (`load_hint: u8`) in `RelayHelloAck` — new field must be additive/version-gated so old relays/clients ignore it (fail-closed default = "no signal, use priority"). Fleet bundle needs no wire change (capacity field already exists, just gets read).
  - CPU/memory: DRR scheduler is O(1) amortized per frame, needs one bounded ring buffer per session (32 frames × 65KB worst-case = up to 2MB per session at max frame size; realistically far smaller since most frames are ~1.4KB MTU-sized — budget ~50KB/session typical) plus a deficit counter (4 bytes) and an active-session rotation list. At `max_total_sessions=4096` this is a non-trivial memory floor (up to hundreds of MB pathological, tens of MB typical) — this MUST be capacity-planned against the Pi's RAM before being taken seriously; a Pi 4 (2-4GB) can likely afford the typical case but the pathological 65KB-frame case needs its own bound (e.g. shrink the per-session queue depth, or cap queued bytes not frame count).
  - New failure mode: queuing adds latency variance where today it's synchronous drop-or-forward — a session's frames can now sit queued rather than being dropped immediately, which changes the relay's behavior under true overload from "some frames silently vanish immediately" to "some frames vanish after a bounded queuing delay." This must be bounded (queue-full still means silent drop, matching today's fail-closed drop semantics — never blocking).
  - New attack surface: the per-relay-hello `load_hint` field, even though relay-authenticated, gives a client (and thus a compromised/malicious node) more visibility into another relay's aggregate load than exists today (currently zero external signal) — this is a minor new information-disclosure surface (aggregate utilization bucket, not per-session data) and should be reviewed against the SecurityMinimumBar before landing, though it discloses less than the existing loopback `/metrics` already exposes to a local operator.
  - Operational complexity: two new tunable knobs (DRR quantum size, load_hint staleness/decay policy) needing sane fail-closed defaults documented in the relay's config surface.

- **Constraint check:**
  1. No custom crypto / no VPN reinvention — PASS. DRR and power-of-two-choices operate on already-authenticated, already-ciphertext UDP frames; WireGuard is untouched; the relay still forwards ciphertext blindly and gains no plaintext visibility.
  2. Default-deny / fail-closed — PASS with a caveat that must hold at implementation time: missing/stale/malformed `load_hint` or fleet `capacity` MUST fall back to today's `max_by_key(priority)` selection (never fail open to an unverified/unvalidated relay), and DRR queue-full MUST silently drop (matching current drop-on-overload semantics), never block or silently escalate capability. This is a design requirement, not yet code, and any implementation that instead treats a missing load signal as "assume healthy" would violate this constraint.
  3. Rust-first, no new runtime dependency category — PASS. Both DRR and power-of-two-choices are pure control-flow/data-structure changes; no new crate class needed beyond maybe a ring-buffer crate already common in the Rust ecosystem (or hand-rolled, consistent with the "no unsafe" workspace policy).
  4. Backend-abstraction boundary — PASS. This is entirely inside `rustynet-relay` (a service, not a domain crate) plus one field addition to `rustynet-control`'s wire types; it does not touch `rustynet-policy`/`rustynet-dns-zone` and does not leak WireGuard types anywhere new.
  5. Raspberry-Pi-class resource target — CONDITIONAL PASS, flagged in Cost/tradeoffs: DRR's per-session queue memory must be capacity-planned against a Pi's RAM (2-4GB) with the existing `max_total_sessions=4096` ceiling; the design must cap total queued bytes across all sessions, not just per-session frame count, or it fails this constraint under adversarial-frame-size load. Power-of-two-choices at N=2-3 relays (the realistic ceiling for a 2-50 node home mesh) is computationally trivial and clearly passes.

- **Incremental build path:**
  - Phase 1 (S): Add `load_hint: u8` field to `RelayHelloAck` wire format + populate it relay-side from existing `active_sessions`/`max_total_sessions` health-snapshot math (already computed for `/healthz`); no selection-logic change yet, purely plumbing + a unit test that an old client ignoring the field still round-trips.
  - Phase 2 (M): Wire `RelayFleet::select_with_policy` as an actual comparator inside `select_runtime_relay_candidate_with_verified_fleet` for the ≥2-relay-candidate case, implementing power-of-two-choices sampling deterministically (fixed RNG seed for test reproducibility) with explicit fallback-to-priority test coverage (missing load_hint, stale fleet bundle, single-relay case all must produce today's exact behavior — regression-pinned).
  - Phase 3 (M): Add bounded per-session ring buffer + deficit-counter fields to `RelaySession`, still forwarding synchronously (no scheduler task yet) — this phase only proves the data structure and queue-full-drop semantics under a live-lab load-test stage with 2+ concurrent sessions of deliberately mismatched traffic rates.
  - Phase 4 (L, needs a design decision): Replace the synchronous forward-on-recv path with the actual DRR scheduler task draining sessions in round-robin-with-deficit order; this is the one phase that changes the relay's core forwarding loop shape and needs explicit sign-off given it reshapes the core forwarding loop (and should re-examine the `Arc<RwLock<RelayTransport>>` granularity while there) — likely also needs to revisit the `Arc<RwLock<RelayTransport>>` granularity (e.g. per-session locks or a lock-free session table) since DRR fairness is undermined if the pre-existing lock itself remains a serialization point.
  - Phase 5 (M): live-lab stage — extend the existing relay-lifecycle live validators (`documents/operations/active/CrossPlatformRoleParityRoadmap_2026-06-22.md` relay-lifecycle track) with a 2-session fairness stage: one session pushes sustained bulk traffic, a second sends small latency-sensitive probes, assert the second session's forwarding latency stays bounded regardless of the first session's rate — this is the falsifiable claim DRR makes and today's lab has no stage that would catch its absence.

- **How you would know it worked / what would falsify it:** A live-lab stage with one relay, one bulk-transfer session and one latency-probe session sharing it: measure probe forwarding latency (recv-to-send-to timestamp delta) under (a) today's code and (b) DRR. Success = probe latency stays within a bounded envelope (e.g. sub-10ms p99) independent of bulk-session send rate; today's code should show probe latency degrading as bulk-session rate increases (lock contention + no ordering guarantee). For part (a), a 3-relay bench with artificially skewed session counts should show power-of-two-choices converging session distribution closer to even than static-priority selection, measurable via the existing `/metrics` active_sessions gauge across the 3 relay processes. Falsification: if the single-lock contention turns out to be negligible at realistic Pi-class throughput (e.g. `cargo bench`/perf trace shows lock wait time is a rounding error next to syscall/copy costs — plausible given the crate's own perf notes claim ~187ns zero-copy forwards), then DRR's added complexity (queue memory, scheduler task, new drop semantics) buys nothing and should be dropped in favor of just fixing the recv-poll-sleep loop's latency floor instead. Similarly, if real-world Rustynet meshes never grow past 1 relay (plausible given the "home server, zero ingress" architecture decision makes a second relay an unusual/advanced deployment), part (a) should be shelved indefinitely — build Phase 1 only if a concrete 2nd-relay use case is committed to.

- **Prior art:** Mitzenmacher, M. (2001), "The Power of Two Choices in Randomized Load Balancing," IEEE TPDS — exponential improvement in max-load bounds from sampling 2 vs 1 random choice. Shreedhar, M. & Varghese, G. (1996), "Efficient Fair Queueing Using Deficit Round Robin," ACM SIGCOMM — O(1) approximate fair queuing without per-flow floating-point bandwidth calculations, designed for exactly this resource-constrained regime. (fq_codel, Nichols & Jacobson et al., 2012, RFC 8290, was considered as an alternative for part (b) — it adds active queue management / bufferbloat control on top of a DRR-like flow-fair core; noted but not adopted here because Rustynet's relay forwards small bounded UDP frames with no queue buildup problem of TCP-bufferbloat's scale, so plain DRR is the simpler correctly-sized tool and fq_codel's extra AQM machinery would be over-engineering for a Pi-class relay).

- **Adversarial verification (independent citation fact-check + constraint review, 2026-07-01):** Citation fact-check **PASS**. Constraint review verdict: **accept-with-fixes** — the largest correction set of this document; the first item is already applied in the text above:
  - The original draft called the `Arc<RwLock<RelayTransport>>` write lock a "confirmed architectural bottleneck"; review against `main.rs:530-604` / `transport.rs:506` showed the write lock covers only the brief in-memory `forward_packet` bookkeeping (the `send_to` runs under a separate read lock on `allocated_sockets`) — corrected above to a hypothesis pending the lock-wait-time trace, with the flat 100 µs poll-sleep floor + tokio wake order named as the more plausible unfairness source.
  - Part (a)'s two-channel design was internally inconsistent: pre-connection ranking can only consume the up-to-300 s-stale fleet-bundle capacity field (the hello-ack load hint requires an already-established session) — materially weaker than classical power-of-two-choices, which assumes a fresh probe per choice; the prior-art claim must carry that gap explicitly.
  - State the aggregate memory ceiling, not just per-session: 32-frame × 65 KB rings at `max_total_sessions`=4096 is ~8 GB pathological — the Pi-class constraint check must gate on the multiplied number and add a global frame-pool cap.
  - Relay load-balancing/fairness is unplanned territory today: the `RelayLoadBalancingAndFailoverPlan.md` title appearing in `FullRepoAnalysis_2026-05-24.md` §6.4 was never written as a file, and grep finds zero ledger hits for DRR/fairness/power-of-two — no competing plan exists.
- **Designer's own weakest point (disclosed):** the cross-relay half is speculative-on-speculative — `RelayFleet::select_with_policy` has sat with zero callers since it was written, which is itself evidence multi-relay demand has not materialized; at exactly 1 relay, part (a) is a no-op by construction and only part (b) fairness carries value.

## FIS-0008: Multi-instance-aware service routing for nas/llm hosting roles

- **Pipeline / area:** Service-hosting roles (D13: `rustynet-nas`, `rustynet-llm-gateway`, `rustynetd::service_access_state` / `service_exposure`, `rustynet-dns-zone`, `rustynet-policy`). Cross-cutting: gossip/soft-state, not the WireGuard dataplane.

- **Current behavior (grounded):** Verified by direct read, not just the grounding notes. `MembershipNode` (`crates/rustynet-control/src/membership.rs:139-148`) carries only `node_id, node_pubkey_hex, owner, status, roles, capabilities, joined_at_unix, updated_at_unix` — no load/health/capacity field of any kind. `DnsZoneRecord` (`crates/rustynet-dns-zone/src/lib.rs:78-90`) is `label -> target_node_id` plus `aliases: Vec<String>` — one label maps to exactly one node id, no priority/weight, no record set. `rustynetd::daemon::materialize_service_access_state` (`crates/rustynetd/src/daemon.rs:~4243`) derives `grants.v1`/`peers.v1`/`scopes.v1` per node from `ContextualPolicySet::evaluate_with_membership`, one snapshot per host — there is no concept of "which of N nas hosts" anywhere in this path. `rustynet_llm_gateway::protocol::Event::HelloOk` (`crates/rustynet-llm-gateway/src/protocol.rs:101-104`) carries `version, models: Vec<String>, tokens_used_in_window: u64` — no queue depth, no tokens-in-flight, no latency estimate. `rustynet_nas::protocol::Response::HelloOk` (`crates/rustynet-nas/src/protocol.rs:91-92`) is the analogous shape for nas (quota/usage only). The peer-gossip channel (`crates/rustynetd/src/peer_gossip.rs:106-114`, `GossipBundle{source_node_id, sequence, timestamp_unix, candidates: CandidateSet, signature}`) is a narrowly-typed, security-reviewed traversal-candidate carrier (`CandidateSet`, capped at `MAX_CANDIDATES_PER_BUNDLE=32`) — it is not a general soft-state bus and should not become one. Finally, per grounding (re-verified via grep, zero hits for `RustyBackup`/`RustyAI` consumer code and zero hits for `queue_depth`/`tokens_in_flight`/`in_flight` anywhere in `rustynet-llm-gateway`): **no client for either service exists in the repo at all** (roadmap M6, future separate program), and the gateway always instantiates `MockEngine` (`crates/rustynet-llm-gateway/src/main.rs:58-59`), so there is no real inference backend whose queue depth could even be measured today.

- **Proposed mechanism:** Two classical, well-understood techniques, kept deliberately separate from each other and from the signed trust plane:
  1. **DNS-style weighted/prioritized record sets (RFC 2782 SRV semantics, without inventing a new DNS record type)** for *service discovery*: extend `rustynet-dns-zone`'s signed zone schema so one label can resolve to a *set* of `(target_node_id, priority: u8, weight: u16)` tuples instead of a single `target_node_id`, mirroring SRV's priority-then-weighted-round-robin selection order. This is a schema extension to an existing owner-signed artifact, not a new mechanism — the zone is already admin-authored and Ed25519-signed; we add cardinality, not a new signer, key, or trust boundary.
  2. **Power-of-two-choices load balancing (Mitzenmacher, 2001, "The Power of Two Choices in Randomized Load Balancing")** for *instance selection among live, policy-permitted candidates*: the client (or, until a client exists, the daemon on the client's behalf as a routing hint) samples 2 of the N candidate instances for a label from the DNS-zone-derived set, and picks the one with the better live signal — least loaded for nas (fewer active sessions / more headroom vs `DEFAULT_QUOTA_LIMIT_BYTES`), and for llm specifically, **least queue-depth / fewest tokens-in-flight** (an M/M/1-adjacent queuing-theory signal: with inference latency dominated by the serving queue rather than network RTT, the correct load proxy is queue occupancy, not RTT). Power-of-two avoids both the herd-to-the-advertised-best-instance failure mode of naive least-loaded-of-N and the O(N) polling cost of checking every instance every time — the textbook reason it beats both random and full-least-loaded at this kind of small-N scale.
  3. **Concrete Rustynet-specific adaptation (the part that isn't a restatement of the technique):**
     - **Signed layer (trust-bearing, admin-signed, goes through the existing bundle path):** `DnsZoneRecord` gains `target_node_id: Vec<(String, u8, u16)>` — priority/weight per candidate node id — populated by the zone owner exactly as today (admin authors the zone), so *which nodes are eligible at all* stays a signed, capacity-holder decision. No automatic promotion into this set — a node is never added because it advertised `serves_nas`; it's added because the human zone author put it there, same trust model as today's single-target records.
     - **Unsigned soft-state layer (liveness/load only, never capability-granting):** a new tiny periodic unsigned UDP advertisement, `ServiceLoadBeacon{node_id, service: Nas|Llm, epoch_of_membership_seen: u64, active_sessions: u16, headroom_pct: u8 (nas) | queue_depth: u16 + tokens_in_flight_est: u32 (llm), ttl_secs: u16}`, broadcast on a **new, separate port** from both the membership-gossip port (51821) and the peer-gossip candidate channel — deliberately NOT folded into `GossipBundle`/`CandidateSet`, because that channel's `MAX_CANDIDATES_PER_BUNDLE`-capped, security-reviewed shape is for traversal candidates and mixing in load telemetry would both bloat a hardened wire format and give unsigned data a ride inside a channel whose whole job is signed-adjacent reachability. The beacon is **unsigned by design** (it is a perf hint, cheap to forge, and forging it can only make an attacker's own instance look *worse*, never grant it capability it doesn't already hold) but is only *meaningful* if the advertising node_id is *already* present in the caller-side default-deny `grants.v1` for that service — a beacon from a node absent from (or revoked from) signed state is dropped unconditionally before it ever reaches selection logic. This is the fail-closed hinge: **the beacon can only reorder among already-authorized instances; it can never add one.**
     - **Selection integration point:** a new pure function in the *client-side* consumer path (M6-gated, see below), `select_service_instance(candidates: &[SignedZoneCandidate], beacons: &BTreeMap<NodeId, LoadBeacon>, grants: &GrantSet) -> Option<NodeId>`, implementing: filter candidates to `grants ∩ zone-set ∩ (beacon present OR beacon stale-tolerant fallback)` → lowest signed `priority` tier → power-of-two among same-tier survivors by (llm: queue_depth ascending, tie-break tokens_in_flight_est; nas: headroom_pct descending) → **return `None` (no selection, caller must fail closed / retry / surface "no healthy instance") if the filtered set is empty, all beacons are stale beyond `ttl_secs`, or ambiguous** (this is the default-deny requirement translated into routing: absence of fresh signal is a routing failure, not a silent fallback to round-robin-blind).

- **Why this is the intelligent version:** At today's actual scale (2-50 nodes, plausibly 1-3 nas/llm instances per mesh even at the high end) the *selection algorithm* itself buys little — with N≤3, power-of-two-choices and full-least-loaded are statistically indistinguishable, and a human could eyeball which nas host is loaded. The genuine value is elsewhere: (a) the **DNS-zone schema extension** is the actual load-bearing design decision, because it's the one piece that's expensive to change after a client exists (a signed-bundle schema bump touches every zone author and every parser); designing it now, even unused, avoids a breaking change later. (b) The **queue-depth-not-RTT** framing for llm is a correctness point worth stating even at N=1: naive health checks (the existing unwired `evaluate_health`/`probe_health`) would still misroute under a single overloaded instance if health is boolean-alive rather than graded-by-load — so even single-instance llm benefits from *reporting* queue depth (for saturation alerting) before multi-instance selection is ever exercised. (c) The win becomes real, not marginal, specifically in the household/small-office topology this repo targets when it *does* cross into 2+ nas/llm instances (e.g., a home-NAS box plus a laptop opportunistically also serving nas while docked, or a beefy desktop GPU box plus a Pi both advertising `llm`) — exactly the "occasionally-connected" heterogeneous-capacity mix the resource profile calls out, where naive round-robin would regularly route to the Pi's queue while the GPU box sits idle.

- **Cost / tradeoffs:** Wire: the signed zone record grows from one `target_node_id` string to a small tuple list — bounded (cap at, say, 8 candidates per label, consistent with `MAX_ALIAS_COUNT`-style caps already in this file) so zone-bundle size stays in the same order of magnitude. The unsigned beacon is tiny (well under 100 bytes) but is a **new periodic broadcast on a new port** — new attack surface (spoofable source, though harmless per the fail-closed hinge above; still needs its own rate-limit/dedup to avoid being an amplification/noise vector) and new operational surface (one more thing that can silently stop working — stale-beacon-as-failure-signal is the right default but means a client-side bug in beacon parsing degrades availability, not just optimality). CPU/memory: negligible (a handful of small periodic UDP sends/receives per host; selection is O(N) over ≤8 candidates). Failure modes: (1) beacon flood/spoofing — mitigated by requiring grants-membership before honoring a beacon, plus a per-source rate limit; (2) beacon staleness causing false "no healthy instance" — mitigated by `ttl_secs` chosen conservatively relative to beacon cadence (e.g., 3x the send interval); (3) priority/weight zone-authoring complexity — a human now has to reason about weights, a genuine new operational burden vs. today's one-line label→node mapping, and for a home-lab operator that's a real UX cost with no upside until they actually run 2+ instances. Operational complexity is the honest downside here: this is machinery for a scale (2+ instances of the same service) that essentially no current Rustynet deployment has reached.

- **Constraint check:**
  1. *No custom crypto/VPN protocol* — PASS. No dataplane change; the beacon rides plain UDP outside the tunnel's crypto path (or inside the tunnel as ordinary overlay traffic — implementation detail, not a new crypto primitive either way); WireGuard is untouched.
  2. *Default-deny / fail-closed, elevated capability stays admin-signed* — PASS by construction: the zone-set membership (which nodes are eligible at all) is unchanged from today's signed-authorship model; the only new automatic decision is *ordering among already-signed-and-granted candidates*, and that decision defaults to `None`/no-route when signal is missing, stale, or ambiguous — it never grants a route to a node absent from `grants.v1` or the signed zone set. This is the one property the design was built around; if reviewed code ever lets a beacon alone create a route to an ungranted node, that's the fail condition (see falsification below).
  3. *Rust-first, no new runtime category* — PASS. Power-of-two-choices and priority/weight ordering are a dozen lines of comparison logic; no ML runtime, no external service.
  4. *Backend-abstraction boundary* — PASS. All of this lives in `rustynet-dns-zone` (schema), a new small client-side selection module (transport-agnostic — it consumes zone records + beacons + grants, never a WireGuard type), and `rustynetd`'s existing service-access materialization path. No backend crate is touched.
  5. *Resource profile (Pi-class anchor, 2-50 nodes)* — PASS with the caveat above: this is explicitly *not* a datacenter mechanism (no consistent hashing ring, no gossip-based failure detector à la SWIM, no vector clocks) — it's the smallest classical toolset (SRV-style static priority/weight + power-of-two sampling) that scales down to N=2 without embarrassment and doesn't bring in machinery that only pays off at N=100+.

- **Incremental build path:**
  - **Phase 1 (S):** Extend `rustynet-dns-zone` schema/tests to accept a `Vec<(target_node_id, priority, weight)>` per record, cap-bounded, with a compatibility path so existing single-target records parse as a one-element priority-0 list (no breaking change to existing zone bundles). Pure data-structure + round-trip test work, no runtime wiring.
  - **Phase 2 (S):** Add `queue_depth`/`tokens_in_flight_est` fields to `rustynet_llm_gateway::protocol::Event::HelloOk` (and equivalent headroom fields to nas's `HelloOk`) as **additive, optional-by-default** wire fields, populated from real state once available (today: `0` from `MockEngine`, honestly labeled as a stub until a real engine adapter exists — this is *reporting*, works even at N=1, and is useful standalone for saturation observability regardless of the rest of this proposal).
  - **Phase 3 (M, dependent on nothing beyond phases 1-2):** Design + implement the unsigned `ServiceLoadBeacon` wire format, its own port, rate-limiting, and the grants-membership-required filter — this is genuinely testable today via unit tests and a live-lab stage that spins up 2 nas or 2 llm instances (the live lab already runs multi-VM topologies) and asserts beacons from a revoked/ungranted node are dropped.
  - **Phase 4 (L, hard-gated on D13/M6 — a real client existing):** The actual `select_service_instance` consumer logic and its live-lab validation (route N requests across 2 real instances, assert convergence toward the less-loaded one, assert fail-closed when beacons go stale) cannot be meaningfully built or proven until the `RustyBackup`/`RustyAI` client program (M6) exists to be the thing making requests — building the selector without a caller is testable in isolation but unproven in the way the live lab proves other stages. This phase should be explicitly deferred and re-scoped when M6 starts, not built speculatively now.

- **How you would know it worked / what would falsify it:** Success signal: a live-lab stage with 2 llm instances of deliberately different simulated load (one fed a busy-loop of concurrent requests) shows client (or routing-hint) traffic statistically preferring the lower-queue-depth instance over N trials, AND a stage that revokes one instance's `serves_llm` capability mid-run shows its beacon immediately stops influencing selection (within one beacon TTL) even if the beacon keeps arriving (i.e., the grants-filter, not the beacon's own content, is what stops routing to it). Falsification: if in practice zone-authored priority/weight is never actually used by real operators (everyone just runs one instance per service, as the current architecture and target scale strongly suggest), this entire mechanism is dead weight that adds an unsigned wire surface for zero realized benefit — the honest tell would be near-zero live-lab or field usage of a >1-candidate zone record after the client program ships, at which point this should be ripped out rather than maintained as unused optionality.

- **Prior art:** RFC 2782 (Gulbrandsen, Vixie, Esibov, 2000) — DNS SRV records, priority/weight selection semantics. Mitzenmacher, "The Power of Two Choices in Randomized Load Balancing" (2001) — the core selection algorithm. Least-connections / least-outstanding-requests load balancing as used in classical L4/L7 balancers (e.g., HAProxy `leastconn`) for the queue-depth-as-load-proxy framing. M/M/1(/k) queuing theory for the "latency is queue-dominated under load" justification specific to inference serving.

- **Adversarial verification (independent citation fact-check + constraint review, 2026-07-01):** Citation fact-check **PASS**. Constraint review verdict: **accept-with-fixes** —
  - Resolve the internal transport inconsistency before Phase 3: the mechanism section puts the load beacon on "a new, separate port" while the cost section calls inside-tunnel-vs-outside an "implementation detail" — these are NOT interchangeable; a new port means new allow-rules on macOS PF / Windows WFP / Linux nft on every node, which the cross-platform parity mandate tracks as real work.
  - Pin the beacon sender/receiver code home to `rustynetd` (the daemon layer); `rustynet-dns-zone` gains only the signed schema extension — stated explicitly so implementation drift cannot violate the transport-agnostic domain-crate boundary.
  - Specify the beacon admission shape now (per-source-node_id token bucket, drop-if-membership-unknown BEFORE the rate-limit check), not as a to-be-designed footnote — for a new unsigned UDP surface this control must have a stated shape before implementation starts.
  - Drop the M/M/1(/k) queuing-theory citation: no actual queuing model (arrival rate, service rate, utilization) is used anywhere in the mechanism; queue-depth as a load proxy for inference serving stands on its own.
  - Phase 1 must include a byte-identical round-trip test proving existing signed zone bundles still verify unchanged (signature-stable migration), since the single-target-record schema decision is called out as irreversible-later.
- **Designer's own weakest point (disclosed):** the entire selection-logic half (least-loaded-of-two, latency EWMA, queue-depth routing) is unreachable until a client exists to consume it — there is genuinely zero client code in the repo today (D13 roadmap M6, confirmed by grep); this is wire-format + policy-integration design for a consumer that does not exist yet, verifiable only by unit/property tests until M6 lands.

## Considered and rejected

Ideas explored during this pass that did not survive the rigor bar (§5 of the research charter) or the hard constraints — recorded so nobody re-proposes them without this context. The first two were taken all the way through design + adversarial review and **rejected by the reviewer**; the rest were rejected during design analysis.

1. **Time-windowed Bloom-filter cascade for enrollment-token single-use enforcement** (fully designed, citation-check PASS, constraint review **REJECT**). The gap it targets — `ConsumedTokenLedger::purge_expired` is a documented no-op stub over a grow-only `HashSet<[u8;16]>` (`crates/rustynetd/src/enrollment_token.rs:361-393`) — is **already tracked as RN-26** in `SecurityReview_2026-05-24.md` (restated in `SecurityAndQualityAudit_2026-06-10.md`), with the exact remediation already prescribed: spool `token_id`+`expires_at`, prune at load. The design's own null hypothesis (exact `HashSet` + `BTreeMap<expiry, token_id>` companion index) dominates the probabilistic structure on every axis at 2-50-node scale — same bounded footprint, zero false positives — while Bloom false positives would force out-of-band token re-mint/redistribution (QR/file), a real availability/UX regression introduced to solve a problem the design itself conceded has no realistic operational horizon. Verdict: build RN-26's exact fix; do not build the cascade.

2. **Multi-anchor ABD/quorum-freshness read for DNS zone bundle fetch** (fully designed, citation-check PASS, constraint review **REJECT** — dispositive). The target path is **dead code in the hardened daemon**: `validate_daemon_config` (`crates/rustynetd/src/daemon.rs` ~L10135) hard-rejects any config with `dns_zone_url` set ("remote network state fetch is disabled in hardened daemon paths; use pinned local signed artifacts") and `StateFetcher::new_from_daemon` unconditionally discards remote URLs — the SR-001 P0 remediation (archived `SecurityReview-2026-03-24.md`), regression-pinned by `validate_daemon_config_rejects_remote_fetch_urls`. Additionally, `AnchorNodeRoleDesign_2026-05-21.md` L358 already considered multi-anchor read coordination for the sibling bundle-pull endpoint and answered no ("clients pick any" — per-artifact signature + freshness verification is sufficient). Any revival must first re-open the SR-001-closed remote-fetch surface (TLS 1.3/mTLS per that review's own recommendation) — a security-review-gated precondition that dwarfs the quorum logic itself.

3. **Auto-election / autonomous role assignment** — violates the hard constraint that an intelligent subsystem may PROPOSE capability placement but never GRANT it (elevated capability stays admin-signed). Only the advisory form survives, as FIS-0005.

4. **Authenticated per-packet outer relay framing as a load-signal channel** — contradicts the recorded `PlugAndPlayTraversalRelayDeltaPlan_2026-03-29.md` §9.4 design decision (v1 relay attribution is by allocated UDP port; no outer framing). Load hints must ride control-plane state instead, which is what FIS-0007 does.

5. **Full BBR congestion control in the userspace pump** — oversized for the target: BBR's bandwidth-delay-product model requires delivery-rate sampling infrastructure whose complexity dwarfs the benefit at home-mesh scale; the right-sized instrument is the RFC-6298-style SRTT/RTTVAR + loss-burst estimator of FIS-0004.

6. **Merkle-tree anti-entropy (Dynamo/Cassandra-style) for gossip** — overkill for this state shape: per-source strictly-monotonic u64 sequences make a 40-byte `(source, sequence)` digest a complete anti-entropy summary at ≤50 nodes; Merkle trees pay off only for large unordered keyspaces. ## FIS-0009: Cross-session NAT-behaviour priors + NAT-aware probe pruning (Happy-Eyeballs history cache + decaying-Bernoulli success estimator)

- **Pipeline / area:** NAT traversal candidate/pair planning (`crates/rustynetd/src/traversal.rs` `plan_direct_probes`/`execute_ice_pair_race`, `crates/rustynetd/src/ice_priority.rs` `generate_candidate_pairs`), plus the stranded runtime NAT classifier in `crates/rustynet-cli/src/ops_cross_network_preflight.rs`.

- **Current behaviour (grounded):**
  Three relevant facts, all verified:
  1. The typed NAT model `NatProfile`/`NatMappingBehavior`/`NatFilteringBehavior` and its predicate `direct_udp_viable(local, remote)` (traversal.rs ~888–930) exist but have **zero non-test callers** — the direct-probe planners `plan_direct_probes` (traversal.rs:1137), `plan_remote_probes` (traversal.rs:1190) and `execute_ice_pair_race` (traversal.rs:1402, driven from phase10.rs:5723) take no `NatProfile` argument and filter candidates only by `source.direct_eligible()` + RFC 8445 priority. A double-symmetric (hopeless) candidate pair is therefore probed for the full `simultaneous_open_rounds` (default 3) before relay fallback, even though `direct_udp_viable` could have predicted failure up front. `[verified: sonnet-5]`
  2. `NatProfile` is only ever *constructed* in test fixtures; it derives `Debug, Clone, Copy, PartialEq, Eq` but **not** `Serialize`/`Deserialize`, and nothing writes a per-peer NAT class or traversal-success record to disk — no NAT/traversal history survives a daemon restart or carries across separate connection attempts to the same peer. The on-disk per-peer record types that DO exist (`PeerRecord` in `ops_peer_store.rs`, `KnownPeerView` in `ops_network_discovery.rs`) carry only identity/endpoint/role, no NAT/history field. `[verified: sonnet-5]`
  3. A *real* runtime NAT classifier already exists — `classify_nat()` → `NatClassHeuristic { cone_nat_likely | port_restricted_likely | symmetric_likely | insufficient_data }` in `crates/rustynet-cli/src/ops_cross_network_preflight.rs:127/355`, using multi-STUN-server comparison (RFC 5780-style). But it is a standalone operator CLI diagnostic (`rustynet ops cross-network-preflight`) that writes a one-shot JSON report and is **never read back by `rustynetd`** — grep for `NatClassHeuristic`/`classify_nat` inside `crates/rustynetd/` is empty. `[verified: sonnet-5]`
  So the codebase contains a working NAT classifier, a NAT-viability predicate, and per-peer disk records — three pieces that are never connected, and the probe planner learns nothing from one connection to the next.

- **Proposed mechanism:**
  Named techniques: **Happy Eyeballs destination-history caching** (RFC 8305 §5 / RFC 6555, Schinazi & Pauly — cache which address family/candidate class actually connected last time and prefer it next time) combined with a **Beta-Bernoulli conjugate success estimator with exponential time decay** (classical Bayesian estimation; the decay makes it a forgetting/【sliding】 estimator so stale priors wash out — the same "recency-weighted success rate" idea TCP Fast Open's cookie cache and ICE's aggressive-nomination heuristics lean on). The two combine into a per-peer **traversal prior** that reorders (never restricts) the existing priority-sorted pair list and prunes only pairs a *confirmed* NAT profile proves hopeless.
  Concrete Rustynet adaptation, three changes, each fail-open:
  1. **New persisted per-peer record** `PeerTraversalPrior` (new serializable struct, lives beside the existing peer store, NOT in a domain crate): `{ peer_node_id, last_success_class: Option<CandidateClass>, per_class: BTreeMap<CandidateClass, ClassStat>, observed_nat: Option<NatProfileSnapshot>, updated_at_unix }`, where `ClassStat { alpha: f32, beta: f32 }` are the Beta pseudo-counts (successes+1, failures+1) and `CandidateClass ∈ {HostV4, HostV6, SrflxV4, SrflxV6}` keys the existing candidate taxonomy. On each probe outcome the daemon updates the winning class's `alpha` and the tried-but-lost classes' `beta`, first applying exponential decay `alpha ← 1 + (alpha-1)·e^(−Δt/τ)` (and same for beta) with a half-life τ ~ 24h so a week-old prior barely counts. Posterior mean success `p̂_class = alpha/(alpha+beta)`. `NatProfileSnapshot` is a `Serialize`-able mirror of `NatProfile` (the derive gap in fact #2 is why a mirror, not `NatProfile` itself, is persisted). `[inference]`
  2. **Reorder, don't restrict, the pair list.** After `ice_priority::generate_candidate_pairs` produces the RFC-8445-priority-sorted pairs inside `execute_ice_pair_race`, apply a *stable* secondary sort by the peer's prior: `effective_rank = ice_priority − λ · logit(p̂_class(pair))` with a small λ so the prior only breaks ties / promotes a historically-winning class a few slots, never overrides a large ICE-priority gap. Happy-Eyeballs shortcut: if `last_success_class` is still present in this connection's candidate set, float exactly one pair of that class to the front of round 0. The RFC 8445 ordering and the deterministic controlling/controlled role assignment are **untouched** — this is a re-rank of an already-valid list, the same discretion the planner already exercises. `[inference]`
  3. **Prune only on a *confirmed* hopeless profile.** Wire the already-built `classify_nat` result into `rustynetd` (persist the local node's own `NatClassHeuristic` after preflight; learn the peer's from observed probe behaviour) and, only when BOTH ends are confidently `symmetric_likely`/hard, call the existing `direct_udp_viable` to **skip the direct race and arm relay immediately** — turning ~240ms + 3 rounds of doomed probing into an instant relay decision. Absent a confident classification (the common/default case) → probe exactly as today (fail-open). `[inference]`

- **Why this is the intelligent version:**
  Today every reconnect to a known peer re-derives the path from a cold ICE race with no memory — a phone that reconnects to the same home anchor 40× a day pays the full multi-round punch dance every time, and a confirmed double-symmetric pair burns three doomed rounds before relay on every single attempt. The win is genuinely marginal for a 2-node all-cone LAN (the first pair wins round 0 regardless) and real at the target's hard end: mobile clients that reconnect constantly across cell/Wi-Fi transitions, and mixed topologies with some symmetric NATs where relay-skip saves both latency and probe traffic. The elegance is that it *reuses three pieces the codebase already built and left disconnected* (the classifier, the viability predicate, the peer store) rather than inventing machinery. `[inference]`

- **Cost / tradeoffs:**
  Wire bytes: **zero** — no new network message; priors are local disk state. Disk/memory: one `PeerTraversalPrior` per known peer (~80–120 bytes), bounded by mesh size (≤50) — a few KB. CPU: an O(1) Beta update + an O(k log k) re-sort of ≤24 pairs per probe — negligible. New failure modes: (a) a **stale/poisoned prior** could mis-rank — bounded because it only re-ranks an already-valid, still-authenticated candidate list (the WireGuard handshake is the real reachability gate; a bad prior costs at most one wasted probe ordering, self-corrects via the decay + the next outcome update), and pruning requires a *confident, both-ends* hopeless classification, not a single sample; (b) a peer that lies to bias its own classification can, at worst, get itself relayed sooner or probed in a different order — never a trust/capability change. Operational: one new persisted file (peer-prior store) and a decay-constant τ + rerank-weight λ to document. `[inference]`

- **Constraint check:**
  1. No custom crypto / no VPN reinvention — **PASS**: reuses existing candidate/pair structs, the existing STUN classifier, and the existing WireGuard handshake as the reachability oracle; adds only local scoring arithmetic. `[inference]`
  2. Default-deny / fail-closed — **PASS**: a missing/stale/low-confidence prior degrades to *exactly today's* behaviour (probe everything in ICE order); pruning fires only on a confident both-ends-hopeless classification and only ever chooses relay (the safe fallback), never grants anything. `[inference]`
  3. Rust-first, no new runtime category — **PASS**: pure Rust (Beta pseudo-counts + exponential decay are float arithmetic); no ML runtime, no external service. `[inference]`
  4. Backend-abstraction boundary — **PASS**: `PeerTraversalPrior` + scoring live in `rustynetd` (daemon layer, alongside the existing peer store); `NatProfile`/`direct_udp_viable` are already in `rustynetd/traversal.rs`; nothing touches `rustynet-control`/`rustynet-policy`/`rustynet-dns-zone`, and the `#![forbid(unsafe_code)]` header is untouched. `[inference]`
  5. Pi-class / 2–50 nodes — **PASS**: KB-scale state, O(1) updates, no probe-bandwidth increase (it *reduces* doomed probing). `[inference]`

- **Incremental build path:**
  - Phase 1 (M): add the serializable `PeerTraversalPrior`/`ClassStat`/`NatProfileSnapshot` types + pure Beta-with-decay update and posterior functions in a new `rustynetd` module, fully unit-tested (given an outcome sequence, assert posterior + decay trajectory). No wiring. `[inference]`
  - Phase 2 (M): persist/load the peer-prior store next to the existing peer store; update it from the existing probe-outcome sites in phase10.rs (`TraversalDecision::Direct`/`Relay` arms, phase10.rs:5738–5776) — record-only, do NOT yet re-rank. Emit the posterior in the status line for observability first (matches the repo's prove-the-signal-before-acting discipline). `[inference]`
  - Phase 3 (M): apply the stable secondary re-rank + Happy-Eyeballs front-float inside `execute_ice_pair_race` behind a config flag defaulted OFF. `[inference]`
  - Phase 4 (L, needs a design decision): wire `classify_nat` into the daemon, persist local NAT class, and enable confident-both-ends relay-skip via `direct_udp_viable`. The design decision: how confidently must a *peer's* NAT be classified from passive probe observation (vs. requiring an explicit RFC-5780 exchange) before we trust it enough to skip the direct race? Flip the Phase-3 flag default only after a live-lab pass. `[inference]`

- **How you'd know it worked / what would falsify it:**
  Positive: a live-lab stage that connects, disconnects, and reconnects a client to the same peer repeatedly under a fixed NAT profile should show time-to-direct-path (or time-to-relay for a symmetric pair) dropping on the 2nd+ attempt vs. the 1st, with fewer probe rounds. Falsification: if in realistic mobile scenarios the peer's candidate set changes so much between reconnects that `last_success_class` is rarely still present, the Happy-Eyeballs half buys nothing; and if double-symmetric pairs are rare enough at real deployments that relay-skip almost never fires, Phase 4 is dead weight — the falsifying result is a live-lab/telemetry run showing <5% of reconnects benefit, in which case ship Phases 1–3 (the cheap re-rank) and drop Phase 4. `[inference]`

- **Prior art:** RFC 8305 Happy Eyeballs v2 (Schinazi & Pauly, 2017) §5 destination-history caching; RFC 6555 (Wing & Yourtchenko, 2012); RFC 5780 (MacDonald & Lowekamp, 2010) NAT behaviour discovery via STUN — the method `classify_nat` already implements; Beta-Bernoulli conjugate priors with exponential forgetting (standard Bayesian sequential estimation; cf. discounted/sliding-window bandit estimators). RFC 8445 §6.1.2 is the ordering this *re-ranks*, not replaces.

- **Sonnet-5 verification note:** three checks (NAT-prior existence, persistence, classifier construction) — all confirmed; the check *changed the proposal*: it surfaced that `classify_nat` already exists as a working-but-stranded runtime classifier, reframing Phase 4 from "build a NAT classifier" to "wire an existing one," and it confirmed `NatProfile` lacks `Serialize` (hence the `NatProfileSnapshot` mirror in mechanism #1).

- **Designer's weakest point (disclosed):** the persisted prior is only as good as candidate-set stability across reconnects; on a peer whose public srflx endpoint churns every reconnect (heavy CGNAT), the per-class prior still helps (class-level, not endpoint-level) but the Happy-Eyeballs exact-endpoint front-float mostly misses. The re-rank is the durable win; the front-float is the softer one.

## FIS-0010: Principled direct↔relay flap damping — decaying failure-intensity estimator + exponential-backoff circuit breaker (wiring an already-present, unused hysteresis primitive)

- **Pipeline / area:** the live relay-vs-direct decision path — `crates/rustynetd/src/daemon.rs` `traversal_probe_due` (5915) + the phase10.rs commit path (`commit_verified_traversal_path_for_peer`, ~5488), and the two **unwired** damping primitives already in the tree (`TraversalSession` counter FSM in traversal.rs; `consider_path_change_for_peer` stability windows in phase10.rs).

- **Current behaviour (grounded):**
  The production relay/direct decision has **no cross-invocation memory of instability**. `execute_ice_pair_race` decides `Direct` purely on a fresh handshake observed within `simultaneous_open_rounds` rounds of a *single* invocation, and on exhaustion calls `relay_or_fail_closed_for_race`, which chooses Relay vs FailClosed solely on `relay_endpoint.is_some()` — the failure counter `relay_switch_after_failures` / `consecutive_direct_failures` is **never referenced** in that path. `[verified: sonnet-5]` Whether to re-run the race at all is `traversal_probe_due` (daemon.rs:5915): Direct → reprobe when the endpoint left the candidate set OR handshake age > freshness (30s); Relay → reprobe at a flat `next_reprobe_unix = now + reprobe_interval` (30s). `TraversalProbeStatus` (daemon.rs ~2018) carries `{remote_node_id, decision, reason, attempts, selected_endpoint, latest_handshake_unix, evaluated_at_unix, next_reprobe_unix}` — **no failure-count / EWMA / backoff field**. `[verified: sonnet-5]`
  Critically, the codebase contains **two** hysteresis/damping primitives and uses **neither** in the live path: (a) `TraversalSession::on_direct_probe_timeout` implements the "N consecutive failures → relay" counter with `relay_switch_after_failures` (default 3), but the entire `TraversalSession` type has **zero non-test callers** — `TraversalSession::new` is never invoked from phase10.rs or daemon.rs. `[verified: sonnet-5]` (b) `consider_path_change_for_peer` (phase10.rs:5415) implements a proper stability-window gate — `direct_stability_window_ms`/`relay_stability_window_ms` (defaults 3000/5000ms), `pending_path_mode`/`pending_since`, `mark_direct_lost`/`mark_direct_recovered` — but the production driver commits path changes **immediately** via `commit_verified_traversal_path_for_peer` → `commit_path_change_for_peer`, bypassing that gate entirely; `mark_direct_lost`/`mark_direct_recovered` also have zero non-test callers. `set_stability_windows` is exercised only by tests. `[verified: sonnet-5]`
  Net effect `[inference]`: a "gray-failure" peer whose direct path completes a handshake, then goes stale within the 30s window, then re-races successfully, then goes stale again, oscillates direct→relay→direct→relay indefinitely — re-programming the WireGuard endpoint each flip — with nothing that says "you've flapped 6 times in 5 minutes, hold on relay for a while."

- **Proposed mechanism:**
  Named techniques: the **Circuit Breaker** pattern (Nygard, *Release It!*, 2007/2018 — after repeated failures, "open" the breaker and stop retrying the flaky path for a cool-down, with a half-open trial before closing) driven by an **exponentially-weighted failure-intensity estimator** with **exponential backoff + decorrelated jitter** (the AWS/`RFC 6298`-lineage backoff discipline). The insight the bare counter misses: a single success must NOT fully reset the trigger, or a flapping path never trips. An EWMA of the failure *indicator* (1 on relay-fallback/stale, 0 on a stable direct interval) captures *rate*, which a consecutive-counter cannot.
  Concrete adaptation — extend the existing (already-present) machinery rather than add a third primitive:
  1. **Add three fields to `TraversalProbeStatus`** (the per-peer state the daemon already keeps): `flap_intensity: f32` (EWMA of the failure indicator, `x ← (1−α)x + α·indicator`, α~0.3), `breaker_open_until_unix: u64` (0 = closed), `backoff_exponent: u8`. `[inference]`
  2. **On each probe outcome**, update `flap_intensity`. When it crosses an open-threshold `θ_open` (e.g. 0.6) → **open the breaker**: set `breaker_open_until_unix = now + base_cooldown · 2^backoff_exponent + jitter` (decorrelated jitter, reusing the CSPRNG-with-fail-soft pattern already in `schedule_proactive_refresh`, traversal.rs:1632), increment `backoff_exponent` (capped). While the breaker is open, `traversal_probe_due` returns *stay on relay* for that peer regardless of the 30s cadence — i.e. stop re-racing a known-flaky direct path. When the cooldown elapses → **half-open**: allow exactly one direct re-race trial; success closes the breaker (decay `flap_intensity`, reset `backoff_exponent` after a sustained-good interval), failure re-opens with a longer cooldown. Below `θ_close`, behaviour is byte-identical to today. `[inference]`
  3. **Reuse the existing stability-window gate.** Route the production commit through `consider_path_change_for_peer` (the already-built, currently-bypassed gate) instead of committing immediately, so a direct↔relay flip must persist for the stability window before it re-programs the endpoint — this alone kills sub-window oscillation and is a small wiring change to an already-tested function. `[inference]`

- **Why this is the intelligent version:**
  A bare consecutive-failure counter (the vestigial `relay_switch_after_failures`) has no notion of *rate*: it fully resets on one success, so the exact pathology that hurts users — a path good enough to occasionally handshake but too flaky to hold — never trips it, and the current live path doesn't even consult the counter. An EWMA-intensity + circuit-breaker converts "is the last attempt failing?" into "how unstable has this path been lately?", and the exponential backoff means a chronically-flaky peer is retried geometrically less often instead of every 30s forever. At 2 nodes on a clean LAN the breaker never opens (intensity stays ~0) so there's zero behaviour change; the payoff is on real lossy links (the "phone on spotty connection" profile). And it's mostly *wiring*: two damping primitives already exist unused — this gives them a principled trigger and connects one to the live path. `[inference]`

- **Cost / tradeoffs:**
  Wire bytes: **zero** (all local per-peer state). Memory: 3 fields (~12 bytes) added to the existing per-peer `TraversalProbeStatus` — trivial at ≤50 peers. CPU: O(1) EWMA per probe. New failure modes: (a) **breaker opens spuriously** on a transient bad patch and holds a genuinely-recovered path on relay for a cooldown — bounded by the half-open trial (it retries, and a good path closes the breaker quickly) and by keeping `θ_open` conservative + `base_cooldown` short; the fail-safe direction is "prefer relay," which is the safe/available choice. (b) **backoff starves a path that only works intermittently but is the only non-relay option** — mitigated by the cap on `backoff_exponent` and by never fully abandoning the half-open trial. Operational: three tunables (α, θ_open/θ_close, base_cooldown) to document alongside the existing `DEFAULT_TRAVERSAL_PROBE_*` family. `[inference]`

- **Constraint check:**
  1. No custom crypto / no VPN reinvention — **PASS**: pure decision-logic over the discrete probe outcomes the daemon already computes; no wire/crypto change. `[inference]`
  2. Default-deny / fail-closed — **PASS**: every uncertain/degraded state biases toward *relay* (available + safe); the breaker only ever *withholds a direct retry*, never grants capability or forces an unvalidated direct endpoint. Below threshold it's identical to today. `[inference]`
  3. Rust-first, no new runtime category — **PASS**: float EWMA + integer backoff; reuses the existing CSPRNG-fail-soft jitter helper. `[inference]`
  4. Backend-abstraction boundary — **PASS**: entirely inside `rustynetd` (daemon + phase10 control logic); driven by discrete `TraversalDecision` outcomes, so it works for **every** backend incl. command-only macOS/Windows (unlike FIS-0001/0004); `#![forbid(unsafe_code)]` untouched. `[inference]`
  5. Pi-class / 2–50 nodes — **PASS**: bytes of state per peer, O(1) math, *reduces* probe frequency on flaky peers. `[inference]`

- **Incremental build path:**
  - Phase 1 (S): add the three fields to `TraversalProbeStatus` + a pure `FlapBreaker` update/decide function (EWMA + threshold + backoff), fully unit-tested against synthetic outcome sequences (assert: steady-good never opens; alternating flip opens; half-open trial closes on sustained success). No wiring. `[inference]`
  - Phase 2 (M): consult the breaker inside `traversal_probe_due` (return "stay relay" while open) behind a config flag defaulted OFF; emit `flap_intensity`/`breaker_state` in the status line first (observe before acting). `[inference]`
  - Phase 3 (M): route the production commit through the existing `consider_path_change_for_peer` stability-window gate instead of the immediate `commit_verified_traversal_path_for_peer` path — the higher-risk change, since it touches the live commit path; gate behind the same flag. `[inference]`
  - Phase 4 (S, after live-lab): flip the flag default ON once a flap-injection stage shows no regression on clean paths. `[inference]`

- **How you'd know it worked / what would falsify it:**
  Positive: a live-lab stage that induces a *gray-failure* direct path (e.g. tc-netem toggling loss on the direct 5-tuple every ~20s) should show the breaker opening after a couple of flaps and the peer settling on relay with geometrically-increasing re-race spacing, instead of today's endless 30s direct↔relay oscillation with endpoint re-programming each flip; a *clean* path must show the breaker never opening (`flap_intensity` stays below θ). Falsification: if on real home networks `flap_intensity` is dominated by benign transient stales and the breaker opens on paths that are actually fine (measured false-open rate high enough to hurt availability), the thresholds can't be tuned to separate signal from noise → shelve in favour of just wiring the stability-window gate (Phase 3 alone) without the intensity trigger. `[inference]`

- **Prior art:** Nygard, *Release It!* (2007/2018) — Circuit Breaker + stability patterns; RFC 6298 (Paxson et al., 2011) exponential backoff lineage; AWS "decorrelated jitter" backoff (Brooker, 2015); EWMA control charts (Roberts, 1959). The consecutive-failure counter this replaces is `TraversalSession::on_direct_probe_timeout` (the design's own weakest baseline).

- **Sonnet-5 verification note:** the check confirmed all four claims AND surfaced the second unwired primitive (`consider_path_change_for_peer` stability windows) I hadn't found — which turned mechanism #3 from "build hysteresis" into "wire the hysteresis that's already there," materially de-risking the proposal.

- **Designer's weakest point (disclosed):** routing the live commit through `consider_path_change_for_peer` (Phase 3) touches the one genuinely load-bearing path in this proposal — the endpoint commit — so it carries real regression risk and must clear the full live-lab handoff suite (exit-handoff, two-hop) before the flag flips; Phases 1–2 are safe/observational, Phase 3 is the one that needs the sign-off.

## FIS-0011: STUN Binding retransmission (RFC 5389/8489 RTO ladder) + parallel-fire budget partitioning for candidate gathering

- **Pipeline / area:** server-reflexive candidate gathering — `crates/rustynetd/src/traversal.rs` `CandidateGatherer::query_stun_server` (290) + `gather_with_observed_at` (258); sibling duplicate impl in `crates/rustynetd/src/stun_client.rs`.

- **Current behaviour (grounded):**
  `CandidateGatherer::query_stun_server` sends the binding request **exactly once** and does a single `recv_from` with a read timeout — **no retransmission ladder** (the tell: a dead `// loop { // removed to silence clippy: loop never loops` comment at traversal.rs:316 shows a retry loop was removed). If either datagram (request out or response back) is lost, that server yields **no** srflx candidate for the entire gather cycle and is only retried on the next caller-driven `gather()`. `[verified: sonnet-5]` Worse, `gather_with_observed_at` computes a **single shared `deadline = now + stun_gather_timeout_ms`** (default 2000ms) across ALL configured STUN servers and passes each server the *full remaining budget* as its per-attempt timeout — so a slow/unresponsive **first** server can consume the whole 2000ms on its one `recv_from`, and every later server is skipped via the `remaining.is_zero()` break before even sending. `[verified: sonnet-5]` There are two independent single-shot STUN implementations — this one and `stun_client.rs`'s `query_stun_server_with_socket` (stun_client.rs:165) — and **neither** has an RTO ladder; traversal candidate-gathering uses the traversal.rs one. `[verified: sonnet-5]`

- **Proposed mechanism:**
  Named technique: **RFC 5389 §7.2.1 / RFC 8489 §6.2.1 STUN retransmission** — retransmit the Binding request on an RTO ladder (`RTO`, `2·RTO`, `4·RTO`, … capped, up to `Rc` attempts, giving up after `Rm·RTO`), which is exactly the reliability mechanism the current single-shot code omits. But applied naively per-server-sequentially it would *worsen* the starvation above. So combine it with **parallel candidate gathering** — the same "fire all, then poll" shape the code already uses in `execute_ice_pair_race` for probe pairs (traversal.rs:1482) — plus **per-server budget partitioning**.
  Concrete adaptation:
  1. **Fire-all-then-collect.** Restructure `gather_with_observed_at` to send Binding requests to **all** configured STUN servers up front (non-blocking sends from the same socket, each with its own transaction id — the code already generates a random 12-byte txn id per query and matches it on parse, traversal.rs:295/486), then enter a single collection loop that `recv_from`s and demultiplexes responses by transaction id until the shared deadline. This removes the "slow server 1 starves servers 2..N" pathology entirely, because all requests are already in flight. `[inference]`
  2. **RTO ladder within the shared budget.** Track per-outstanding-request next-retransmit times; retransmit any still-unanswered request when its RTO expires (start RTO ~250–500ms, double each retransmit, cap `Rc`≈3 within the 2000ms budget — deliberately fewer than RFC 5389's 7 because the gather budget is short and this is candidate discovery, not a control transaction). A single lost datagram now recovers within one RTO instead of costing the whole cycle. `[inference]`
  3. **Keep the wire/parse code identical.** `build_stun_binding_request` and `parse_stun_xor_mapped_address` are unchanged (both already correct + fuzz-tested); only the send/collect scheduling around them changes. Optionally apply the same restructure to `stun_client.rs`'s sibling path (or, better, converge both on one shared gatherer — noted as a follow-up, not required for the core win). `[inference]`

- **Why this is the intelligent version:**
  On a clean link the current single-shot works fine — but the target profile explicitly includes "a phone on a spotty connection," and on a lossy cellular uplink a single dropped STUN datagram currently means **zero server-reflexive candidate for a full gather cycle**, which cascades into no srflx pair, a failed/again-single-shot direct race, and relay fallback — a large reliability cliff from one lost packet. The RTO ladder is the standardized, decades-proven fix (every real ICE stack has it), and folding in parallel-fire + budget partitioning simultaneously repairs the separate, verified starvation bug where one slow STUN server blanks all the others. This is the highest-certainty, most-clearly-correct of the three proposals: it's implementing a mandatory part of the STUN spec the code currently skips. `[inference]`

- **Cost / tradeoffs:**
  Wire bytes: **bounded increase** — at most `Rc`≈3 requests per server per gather instead of 1, and only for servers that don't answer on the first try (a responsive server still answers after one send; no extra bytes on the happy path). At 20-byte requests to a handful of STUN servers, single-digit KB worst case per gather, human-triggered-frequency. CPU/memory: a small per-outstanding-request table (txn id → next-RTO) during the ~2s gather, freed after — trivial. New failure modes: (a) **retransmit amplification** toward a STUN server — bounded hard by `Rc` and the shared deadline; this is standard STUN client behaviour, not novel load; (b) the parallel restructure changes the shape of a load-bearing gather function, so it must preserve the existing filter/dedup semantics exactly (the existing `candidate_gatherer_query_and_timeout_and_filter_and_dedup` test guards `collect_gathered_candidates`, which is unchanged). Operational: two tunables (initial RTO, `Rc`). `[inference]`

- **Constraint check:**
  1. No custom crypto / no VPN reinvention — **PASS**: implements a *standard* part of RFC 5389/8489 the code omitted; STUN is already the gather protocol here; no crypto, no WireGuard-internal change. `[inference]`
  2. Default-deny / fail-closed — **PASS**: retransmission only affects *candidate discovery*; failing to gather an srflx candidate still falls back exactly as today (fewer candidates → relay), and the WireGuard handshake remains the authenticated reachability gate. More candidates can only *improve* direct-path odds, never grant trust. `[inference]`
  3. Rust-first, no new runtime category — **PASS**: `std::net::UdpSocket` + timers, pure Rust, reuses the existing CSPRNG for txn ids; `#![forbid(unsafe_code)]` untouched. `[inference]`
  4. Backend-abstraction boundary — **PASS**: entirely within `rustynetd/traversal.rs` candidate gathering; no domain-crate or backend-API change. `[inference]`
  5. Pi-class / 2–50 nodes — **PASS**: a handful of extra small datagrams only on loss; explicitly sized *down* from RFC 5389's full ladder for the short gather budget. `[inference]`

- **Incremental build path:**
  - Phase 1 (S): introduce a mockable socket seam (trait over `send_to`/`recv_from`, or inject a `UdpSocket`-like) so `query_stun_server`/`gather` become unit-testable — today no test drives them against a socket (only `collect_gathered_candidates` is tested). Add a test harness. `[inference]`
  - Phase 2 (M): implement the fire-all-then-collect restructure of `gather_with_observed_at` with per-server budget partitioning (still single send each) — fixes the starvation bug first, test that a slow server 1 no longer blanks servers 2..N. `[inference]`
  - Phase 3 (M): add the RTO retransmission ladder within the collect loop; new tests over the mock socket assert a dropped first response triggers a retransmit and still yields the candidate within budget, and that a fully-dark server gives up after `Rc` without blowing the deadline. `[inference]`
  - Phase 4 (S): optionally converge `stun_client.rs` onto the same gatherer (or port the ladder), removing the duplicate single-shot impl. `[inference]`

- **How you'd know it worked / what would falsify it:**
  Positive: a live-lab (or bench) stage that drops a configurable fraction of STUN request/response datagrams should show srflx-candidate yield staying high (recovered by retransmit) where the single-shot baseline collapses to near-zero at moderate loss; and a stage with one deliberately-blackholed STUN server should show the *other* servers still producing candidates (starvation fixed). Falsification: if measured real-world STUN loss to typical public servers is negligible AND deployments almost always configure a single STUN server (so parallel-fire/partitioning is moot), the win shrinks to just the retransmit-on-loss edge case — still correct, but lower priority than framed; the falsifying signal is a telemetry run showing srflx first-attempt success already >99% on the target links. `[inference]`

- **Prior art:** RFC 5389 §7.2.1 (Rosenberg et al., 2008) and RFC 8489 §6.2.1 (Petit-Huguenin et al., 2020) — STUN transaction retransmission (RTO, RTO doubling, `Rc`/`Rm` caps); RFC 8445 §14 ICE pacing; the "fire all then poll" pattern is the same one already in this file's `execute_ice_pair_race` (traversal.rs:1482), so the restructure is internally idiomatic.

- **Sonnet-5 verification note:** confirmed single-shot + the dead `// loop {` tell, confirmed two independent single-shot STUN impls, and surfaced the shared-deadline **starvation** bug (per-server timeout = full remaining budget) — which is why mechanism #1 pairs budget-partitioning with the retransmit ladder rather than adding a naive per-server sequential ladder that would have made starvation worse.

- **Designer's weakest point (disclosed):** the highest-certainty proposal but also the one whose payoff most depends on deployment reality — if operators run one nearby, reliable STUN server the loss-recovery win is small; the starvation fix only matters with ≥2 configured servers. It's "obviously correct spec-compliance" more than "clever," which is exactly why it's ranked third despite the lowest risk.

## FIS-0012: Two-budget deficit-round-robin fair drain (FQ-DRR) with lazy engagement for the shared pump loops

- **Pipeline / area:** dataplane pump service order — `RuntimeState::poll_authoritative_socket_with` (generic runtime.rs:699) and `poll_tun_device_with` (:731), plus the macOS mirrors (`poll_authoritative_socket_with_budget`, macOS runtime.rs:776; `poll_tun_device_with`, :825).

- **Current behavior (grounded):**
  One UDP socket and one TUN device are shared across every configured peer. Each tick, `run_worker` (generic runtime.rs:947–985; macOS mirror at :1063, same commands-first → poll socket → poll TUN → expire-round-trip shape `[verified: sonnet-5]`) drains up to `MAX_AUTHORITATIVE_DATAGRAMS_PER_TICK` = 64 datagrams (:23) and 64 TUN packets (:24) in flat `for _ in 0..64` loops, processing strictly in the order the OS handed them back. **There is no per-peer bookkeeping of any kind in either drain loop or its callers — no counters, no queues, no round-robin pointer; the only per-peer state in `RuntimeState` is `peers: BTreeMap<NodeId, PeerConfig>` (config, not scheduling)** `[verified: sonnet-5]`. Each iteration is fully synchronous on a single shared scratch buffer: `try_recv_into(scratch)` → round-trip-response check → `engine.process_inbound_ciphertext(remote_addr, …)` → `apply_engine_processing_outcome`, and the payload slice `&scratch[..len]` is only valid within its own iteration — **deferring a packet past its iteration requires copying it out of scratch; no such copy exists today** `[verified: sonnet-5]`. When a pass consumes its full budget, `udp_budget_exhausted`/`tun_budget_exhausted` (:467–468) make `next_wait_timeout` return `Duration::ZERO` (:668–688) so the worker re-polls immediately — identical semantics in both files `[verified: sonnet-5]`. Peer identity is resolved only *inside* the engine: inbound via `find_node_id_by_endpoint` (engine.rs:372, linear scan over `peer_states` matching `endpoint == remote_addr`; no O(1) endpoint index exists `[verified: sonnet-5]`), outbound via `Tunn::dst_address(packet)` → the **private** `select_peer_for_destination` (engine.rs:380, linear allowed-ips scan; no pub method exposes destination resolution without processing `[verified: sonnet-5]`). The macOS runtime reuses this same shared `engine.rs` — only runtime/tun/socket files are mirrored per-OS `[verified: sonnet-5]`. Known drift between the mirrors: macOS has a UDP-side `poll_authoritative_socket_with_budget(scratch, budget)` test seam + `#[cfg(test)] poll_authoritative_socket_budget_for_test` (macOS :776/:808) that the generic file lacks (budget hardcoded in the loop); the TUN side is hardcoded in both `[verified: sonnet-5]`.
  **Consequence** `[inference]`: which of the (possibly ≫64) waiting packets gets one of the 64 slots this tick is decided purely by kernel-FIFO delivery order. A peer flooding the shared socket occupies slots and buffer space in proportion to its arrival rate; a light peer's keepalive/handshake sits behind the flood. Nothing remembers, across ticks, that a peer has been crowded out.
  **Boundary honesty** `[inference]`: the OS receive buffer sits in front of `try_recv_into`. Packets dropped by the kernel before the loop runs are unrecoverable — no userspace scheduler can un-drop them, and the loop cannot reorder what is still queued inside the kernel. What this proposal controls is (a) service order among packets that DID reach userspace and (b) *where overflow drops happen* under sustained flood: today they happen in the shared kernel FIFO (loss falls on whoever's packets arrive while it's full — no isolation); with per-peer bounded stashes and a read-ahead budget, the loop drains the kernel faster than it processes and overflow moves into userspace queues where each flow's excess is dropped against *that flow's own* cap. Relocate + fair-ify the drop point — not eliminate drops.

- **Proposed mechanism:**
  Named algorithms: **Deficit Round Robin** (Shreedhar & Varghese, SIGCOMM '95 / ToN '96) — the O(1)-amortized max-min-fair scheduler used in real switch egress silicon — for service order, plus the **flow-isolation half of FQ-CoDel** (per-flow queues with per-flow overflow drop; McKenney SFQ 1990 lineage) — deliberately *without* CoDel's sojourn-time control law (rejected below). Adapted to a per-tick-budgeted, pull-based, two-direction, two-file setting:
  1. **Two budgets.** Keep the process budget `B = 64` exactly as today (the tested ceiling — the expensive work bound: decapsulate/encapsulate per packet dwarfs a recv syscall). Add a fixed read budget `R` (default `4×B = 256`) used **only under contention**: drain up to R datagrams per pass into per-flow stashes (cheap `try_recv_into` + memcpy), then *process* exactly ≤ B of them chosen by DRR. Both budgets are fixed constants → a tick's total work stays bounded (256 recvs + 64 process + ≤256 copies worst case), satisfying the per-tick-ceiling ground rule; R exists because with R = B the scheduler has nothing to choose between — every packet read is processed the same tick, so reordering is a no-op `[inference]`.
  2. **Lazy engagement (the N=2–3 answer).** The scheduler is bypassed entirely when the previous pass did not exhaust its budget AND all stashes are empty — the fast path is byte-identical to today's loop (no peer lookup, no copy, no DRR state touched); the added cost at typical 2–3-peer load is two boolean checks per pass. DRR engages only when `udp_budget_exhausted`/`tun_budget_exhausted` was set (the existing backlog signal, reused as the contention detector) or stashes hold carryover `[inference]`.
  3. **Classification = routing metadata only.** Inbound: `remote_addr → FlowKey::Peer(node_id)` via the same endpoint-match predicate the engine uses (linear scan today — the known, already-documented optimization item; at contention time and ≤50 peers this is ≤50 pointer compares per datagram, noted as a dependency, not fixed here `[inference]`). Datagrams from unknown endpoints (handshake initiations, junk) go to one shared `FlowKey::Unclassified` queue that participates in DRR like any flow — which is a quiet security *improvement*: a spoofed-source flood isolates into that one queue's cap instead of crowding every peer `[inference]`. Outbound: `Tunn::dst_address(packet)` → destination peer, via a new `pub(crate) fn resolve_destination_peer(&self, packet: &[u8]) -> Option<NodeId>` wrapping the currently-private `select_peer_for_destination` (verified private today; the wrapper is metadata-only, no processing) `[verified: sonnet-5]`.
  4. **Classic DRR service.** Per flow: `deficit_bytes: usize`, FIFO stash. Each service round, visit the active list; on a flow's turn add `quantum` (default 1500 B = one MTU) to its deficit; serve its head packets while `deficit ≥ packet_len` and global process budget remains, decrementing per packet served; a flow whose stash empties leaves the active list and its deficit resets to 0 (Shreedhar–Varghese anti-hoarding). Weighted variant: `quantum_i = weight_i × 1500` with a per-peer operator knob (plain config field, default 1.0) — same scheduler, no structural change `[inference]`.
  5. **Bounded stashes + pooled buffers.** Per-flow cap 16 packets / 32 KB, global cap = R (256). Overflow tail-drops within the offending flow. Buffers come from a preallocated pool of 2 KB slots (≈ MTU-sized; datagrams > 2 KB — rare, jumbo-frame only — take a one-off heap allocation, honestly counted against the micro-perf discipline this crate maintains). Worst-case pool memory ≈ 512 KB, configurable down for Pi-class nodes; zero when the feature is off or uncontended `[inference]`.
  6. **Semantics preserved.** The round-trip-response check (`try_deliver_round_trip_response`) runs at *read* time exactly as today (it must inspect the datagram anyway, and round-trip replies are control-plane, not schedulable load). Once a packet is *picked*, processing is unchanged: same `process_inbound_ciphertext`/`inject_plaintext_packet` calls, same `EngineProcessingOutcome` handling. Only the picking order changes `[inference]`.
  7. **Both files.** The scheduler itself lands once in `userspace_shared/` (shared module, like `engine.rs`); the generic runtime first gains the same `_with_budget` seam macOS already has (harmonizing the verified drift), then both `poll_*` families gain the identical two-stage wiring. macOS TUN side gets a `poll_tun_device_with_budget` seam at the same time, closing the remaining asymmetry `[inference]`.

- **Why it's actually better (and the measurement plan):**
  Honest framing: **whether this matters in practice is currently unmeasured — no bench or test anywhere in the crate constructs an asymmetric multi-peer scenario** (`benches/dataplane_engine.rs` and `examples/perfprobe_engine.rs` both drive a single peer pair `[verified: sonnet-5]`). So the proposal is sequenced measurement-first: build the probe, quantify the starvation, and let the numbers gate the wiring. The theoretical case `[inference]`: under a 10:1 flood the light peer's handshake/keepalive competes for 64 FIFO slots and kernel buffer space at 10:1 odds — DRR gives it a full quantum every round regardless (max-min fairness), and per-flow caps convert flood-time kernel-FIFO loss (which falls on everyone proportionally) into flood-flow-only loss. At 2–3 peers with no contention the mechanism literally does not engage. The payoff grows with peer count and traffic asymmetry — exit/relay/anchor nodes serving mixed bulk + interactive peers are the beneficiaries.
  **Measurement plan** (extends existing tooling, no new frameworks): (a) new example `perfprobe_fair_drain.rs` following the `perfprobe_engine.rs` fixed-work pattern (WARMUP then MEASURED ops, `Instant` timing, alloc-meter delta `[verified: sonnet-5]`): N=2 and N=8 peers, peer A offered 10× peer B's rate through a scripted-arrival mock socket, report per-peer service share and B's p50/p95 ticks-until-serviced, scheduler off vs on; (b) extend `benches/dataplane_engine.rs` (criterion, `--features test-harness` `[verified: sonnet-5]`) with a fast-path overhead bench (single peer, uncontended, scheduler compiled in but disengaged) and a contended-bookkeeping bench (ns/packet of DRR machinery at N=3 and N=50).

- **Cost / tradeoffs:**
  Memory: scheduler struct + pool ≈ 512 KB worst case (bounded, preallocated, config-shrinkable); zero when off. CPU: fast path +2 branches/pass; contended path per stashed packet ≈ one ≤2 KB memcpy (~50–100 ns) + linear peer lookup (≤50 compares) + O(1) DRR bookkeeping — single-digit-% overhead relative to the ~µs-scale decapsulate each packet costs anyway, and only while contended `[inference]`. Complexity: the real cost. Today's loop is 20 lines and trivially auditable; this adds a scheduler module (~300 lines + tests), a second budget, and a bypass condition — mitigated by keeping the scheduler a pure, separately-tested data structure and the fast path textually identical to today's loop. New failure modes: (a) a bug in stash/pool lifecycle could leak or double-serve a packet — pure-unit-tested module + property tests before any wiring; (b) misclassification (endpoint moved mid-tick) sends a packet to the wrong queue — worst case is unfair ordering for one tick, never loss of validity (processing is unchanged); (c) R read-ahead under flood spends syscall time on packets that may be dropped at the stash cap — bounded by R, and it is exactly the work that buys drop isolation `[inference]`.

- **Ground-rules check:**
  1. Per-tick ceiling preserved — **PASS**: B stays 64 (the two Linux budget tests keep passing unmodified — they assert capped-drain via `.len()` counts, and with the feature off the path is byte-identical; with it on, ≤64 processed per pass still holds); R is a second *fixed* constant, not an unbounded queue `[verified: sonnet-5 — the budget tests assert lengths, not order]`.
  2. Picking order only — **PASS**: packet handling after selection is unchanged; classification uses `remote_addr`/`dst_address` routing metadata only, available pre-processing `[inference]`.
  3. Safe Rust — **PASS**: zero `unsafe` in all three files today `[verified: sonnet-5]` and the design needs none (Vec-pool + BTreeMap/VecDeque) `[inference]`.
  4. Both runtimes — **PASS**: shared scheduler module + explicit per-file wiring plan (and it *reduces* drift by porting the `_with_budget` seam to the generic file and to both TUN sides) `[inference]`.
  5. Small-scale honesty — **PASS**: lazy engagement makes the N=2–3 uncontended cost two branches; the fast-path criterion bench makes any regression measurable before the default flips `[inference]`.

- **What would falsify it:** the perfprobe scenario showing the light peer's p95 ticks-until-serviced improves by less than ~2 worker passes under a 10:1 flood (i.e., kernel-FIFO order was never the binding constraint — e.g. because kernel-buffer drops dominate before userspace ever sees the flood), or the fast-path criterion bench showing >2% throughput regression on the uncontended single-peer path (the common case paying for the rare one). Either result → keep the measurement harness, shelve the scheduler `[inference]`.

- **Prior art:** Shreedhar & Varghese, "Efficient Fair Queuing Using Deficit Round-Robin" (SIGCOMM 1995 / IEEE ToN 1996); McKenney, "Stochastic Fairness Queueing" (INFOCOM 1990); Høiland-Jørgensen et al., "The Flow Queue CoDel Packet Scheduler and AQM" (RFC 8290, 2018) — the per-flow-isolation half only; Demers, Keshav & Shenker, "Analysis and Simulation of a Fair Queueing Algorithm" (SIGCOMM 1989) as the WFQ baseline DRR approximates at O(1).

- **Recommended patch implementation plan:**
  - **New shared module** `crates/rustynet-backend-wireguard/src/userspace_shared/fair_drain.rs` (used by both runtimes, like `engine.rs`):

    ```rust
    pub(crate) struct StashedPacket { buf: PooledBuf, len: usize, remote_addr: Option<SocketAddr> }
    struct FlowQueue { deficit_bytes: usize, packets: VecDeque<StashedPacket>, stashed_bytes: usize }
    pub(crate) struct FairDrainScheduler {
        flows: BTreeMap<FlowKey, FlowQueue>,
        active: VecDeque<FlowKey>,          // DRR active list
        pool: BufferPool,                    // Vec<Box<[u8; 2048]>> free list + oversize fallback
        quantum_bytes: usize,                // default 1500
        per_flow_cap_packets: usize,         // default 16
        global_cap_packets: usize,           // default 256 (= R)
        total_stashed: usize,
    }
    impl FairDrainScheduler {
        pub(crate) fn stash(&mut self, key: FlowKey, payload: &[u8], remote_addr: Option<SocketAddr>) -> StashOutcome; // copies out of scratch; tail-drops within `key` on cap
        pub(crate) fn next_to_process(&mut self) -> Option<StashedPacket>;  // DRR: quantum on round entry, serve while deficit >= len, reset+deactivate on empty
        pub(crate) fn is_empty(&self) -> bool;
    }
    ```
    Core loop sketch (inbound; TUN identical with `FlowKey::Peer(dst)` and no `remote_addr`):
    ```rust
    // fast path: yesterday's loop, untouched
    if !self.udp_budget_exhausted && self.fair_drain.is_empty() { return self.poll_authoritative_socket_flat(scratch); }
    // stage 1: read up to R, classify, stash (round-trip replies delivered inline as today)
    for _ in 0..READ_BUDGET { let Some((len, ra)) = socket.try_recv_into(scratch)? else { break };
        if self.try_deliver_round_trip_response(ra, &scratch[..len])? { continue; }
        let key = self.engine.flow_key_for_remote(ra);            // linear endpoint match, metadata only
        self.fair_drain.stash(key, &scratch[..len], Some(ra));
    }
    // stage 2: process exactly <= B in DRR order
    let mut processed = 0;
    while processed < MAX_AUTHORITATIVE_DATAGRAMS_PER_TICK { let Some(pkt) = self.fair_drain.next_to_process() else { break };
        let outcome = self.engine.process_inbound_ciphertext(pkt.remote_addr.unwrap(), local, pkt.payload(), gen)?;
        self.apply_engine_processing_outcome(outcome)?; processed += 1;
    }
    self.udp_budget_exhausted = processed == MAX_AUTHORITATIVE_DATAGRAMS_PER_TICK;
    ```

## Focused pass — Considered and rejected (dataplane scheduling)

- **CoDel/FQ-CoDel sojourn-time control law** — target-delay control (5 ms) is mismatched to a 10 ms-tick pull loop, and the bounded per-flow stash already caps queueing delay structurally (≤ cap/service-share); per-packet timestamps kept only as an optional debug metric. Isolation half retained, control law dropped. `[inference]`
- **WFQ / virtual-clock per-packet timestamp scheduling** — O(log n) heap + virtual-time arithmetic per packet, strictly dominated by DRR's O(1) at ≤50 flows; approximating WFQ at O(1) is literally DRR's founding contribution. `[inference]`
- **Linux qdisc (fq_codel on the rustynet interface) instead of in-loop scheduling** — zero-code for the TUN direction on Linux only: no macOS/Windows parity, nothing for the inbound UDP direction, invisible to the test suite. Worth noting as an ops-level complement, rejected as the mechanism. `[inference]`
- **Raising the process budget B** — violates the per-tick-ceiling ground rule and doesn't change ordering, only moves the cliff. `[inference]`
- **Per-peer sockets (SO_REUSEPORT et al.)** — would let the kernel do the queue separation, but breaks the single-authoritative-socket transport identity that the daemon builds its socket-identity state around; far out of scope for a scheduling pass. `[inference]`
